# 太仓快递经营体系：账务科目 · 卡片流价格成本 · 阿米巴报表 · OA替代　四项分析

> 基于现有代码（STOTOP.Module.Finance / CardFlow / Express）与 `Taicang/` 历史数据（申通/韵达/极兔总部交易、出港运单、费用报销、派件、报价成本测算、目标经营报表）整理。
> 数据为采样/摘要口径（业务流水未做全量聚合）。日期：2026-06-15。本文为方案与待确认清单，未改动任何代码。

## 执行摘要（先看这 6 条）

1. **科目体系已相当完善、且为快递定制，不需要推倒重来。** 现账套（账套2）已 seed 500 个科目：`1012 其他货币资金`下 31 个"申通总部-XX"预付款明细、`5001 主营业务收入`9 个出港/进港/到付/返利细目、`5401 主营业务成本`23 个面单/中转/派费/操作/运输细目、`5601 销售费用`41 个、`5602 管理费用`35 个。结论：**保留为主，少量新增/调整**。

2. **账务最大缺口 = 韵达、极兔交易完全没有"交易→账套科目"映射。** 申通已有两套映射（`总部映射关系-新`154个编码、`凭证对应表-网点管家`105个编码）；但韵达（95 个三级收费科目）、极兔（164 个费用子类型）**零映射**。这是凭证规则补充工作量的大头。

3. **自动凭证引擎已经存在，且不弱。** CardFlow 里有 `AutoVoucherMatchingEngineV2`（规则匹配）+ `AutoVoucherAuxiliaryResolver`（辅助核算解析）+ `AutoVoucherPlugin`，配合 Finance 的 `FIN凭证规则/FIN凭证手动规则`。优化方向是**喂规则、补多源、做幂等**，不是重写。

4. **价格/成本应做成"运单逐票"的双轨引擎。** 现有报价是"基础矩阵(首续重)＋目的地加收＋一口价"三层，成本测算表是"10 项成本 × 网点 × 目的地 × 重量段"。建议逐票算：收入(报价)−标准成本(测算表)＝标准毛利，再与运单实带的"总部实际结算值"对账成实际毛利。同一份逐票结果同时喂自动凭证与阿米巴，**一处取数、避免口径漂移**。

5. **阿米巴目标报表口径非常明确，可直接复现。** 目标即 `网点梳理经营报表.xlsx` 的"汇总表"：出港/进港**双段**，每段 收入–成本–分摊–利润，且每行都带**单票**列与"取自总部帐单 / 取自网点经营报表 / 根据发件重算"的取数来源。现有阿米巴模型（损益模板+取数规则+分摊+手工项）需从"单一凭证取数"演进到"**多源归集 + 单票口径 + 月度环比**"。下文用城区公司真实数字验证了可复现性。

6. **用卡片流替代钉钉OA可行，且自动记账规则已基本就绪。** 费用报销 41 条凭证规则已覆盖 36/47 类费用；钉钉导出字段、审批流要素、成本中心(费用归属单元)都清楚。建议两步走：先用卡片流**导入**钉钉数据打通自动凭证，再自建"报销卡片(表单+审批+付款+凭证)"彻底替代。

## 1.2 科目体系建议（保留为主）

整体保留现有科目，做以下**针对性调整**（具体编码待确认，见 1.5）：

1. **预付款账户改为"按品牌+网点"成体系**：现有 `1012` 下只有申通(且是石家庄网点)。建议：
   - `101201 其他货币资金—申通预付款`（下设 太仓城区/南郊/沙溪/浏河 明细）
   - 新增 `101202 其他货币资金—韵达预付款`（通达/浏河）
   - 新增 `101203 其他货币资金—极兔预付款`（太仓四区等）
   - 三家网点交易的"扣款/充值/返还"都先过各自预付款账户，与总部对账。
2. **主营收入补 2 个细目**：`5001` 下补 `增值业务收入`（菜鸟裹裹/裹裹寄件/申鲜尊享/到付手续费）、`代收货款手续费收入`；并明确 `500103 进港派送费收入` 同时承接申通/韵达/极兔的派费收入（用辅助核算区分品牌）。
3. **主营成本补"集包费、环保袋、客服理赔"**：测算表与经营报表都有 `集包费`、`环保袋费用`、`私了三大件理赔`，但 `5401` 下未单列。建议在 `540103 出港网络成本` 增 `集包费`，`5401` 增 `客服理赔成本`/`环保袋成本`（或归 `540199`）。
4. **韵达/极兔的"质量考核/网管/仲裁/虚假签收"扣款** → 统一归 `540105 总部考核类罚款`（按品牌辅助核算），不必为每个细类设科目。
5. **资金类动作不入损益**：`调账(0401)`、`取款(0301)`、`转账充值(0201)`、韵达`余额划转/提现`、极兔`提现/往来款/保证金` → 只动 `1012 预付款` 与银行/往来，**不进收入成本**（这是金额最大的几类，必须排除，否则虚增收入成本，见 1.4 待确认）。

## 1.3 辅助核算维度建议

把现在"一个部门维度"拆成**4 个正交维度**（凭证分录的 `F辅助核算JSON` 可多挂）：

| 维度 | 取值 | 挂哪些科目 |
|---|---|---|
| **经营单元/成本中心**（现有，保留） | 城区/南郊/沙溪/浏河 × 申通/韵达/极兔 + 市场部/车队/操作组/业务支持 | 收入、成本、期间费用、折旧、预付款 |
| **品牌**（建议新增，便于按快递公司汇总） | 申通 / 韵达 / 极兔 | 收入、成本、预付款 |
| **客户**（现有，扩展） | 月结客户/父客户/店铺 | 应收、月结收入 |
| **业务员/承包区**（建议新增，供人效与派费分析） | 业务员编码、承包区 | 进港派费成本、揽件派费 |

> 经营单元 + 品牌 两维联用，就能直接驱动阿米巴"按网点、按品牌"两种视角的损益，无需再建一套维度。

## 1.5 待确认清单（账务，编号 D1-x）

- **D1-1（高）资金类动作是否一律不入损益**：`调账/取款/转账充值`、韵达`划转/提现`、极兔`提现/往来款/保证金/充值`金额巨大（仅申通调账就 490万、取款 299万）。建议：仅做 `预付款↔银行/往来` 资金分录，不进收入成本。**默认：是，不入损益。**
- **D1-2（高）三家交易凭证的汇总粒度**：逐笔生成（凭证量巨大）还是"按网点×记账日×费用科目"汇总成一张？**默认：按 网点×记账日 汇总，明细留底。**
- **D1-3（高）申通两套映射以哪套为准**：`总部映射关系-新`（借贷完整）vs `凭证对应表-网点管家`（贷方常缺）。**默认：以"总部映射关系-新"为权威，网点管家表仅作辅助核算补充。**
- **D1-4（高）韵达/极兔符号语义**：韵达"扣款为正"、极兔"扣款为负"，解析时必须各自归一到"收入(+)/成本(−)"再记账。**需确认每家的正负含义表。**

# 二、卡片流 + 价格计算 + 成本计算 优化

## 2.1 现状

- **价格（对客户收入）**：三层 = 基础报价矩阵(首重+续重 `首重价 + G*单价 + (G-3)*…`，按目的地区域/省份、10 个重量段) ＋ 目的地省/市加收 ＋ 一口价(`原一口价`分段表/`一口价结算`明细)。报价数据模型在 `EXP快递报价*` 系列表（主表/矩阵明细/出港加收/作用域/目的地）。运单靠 `报价ID/父客户/目的省份/目的城市/结算重量/付费方式` 命中。
- **成本（对总部支出）**：`太仓系统成本测算` 是 10 项成本（①面单②全网出港费(含加收)③出港操作费④中转费⑤出港派费⑥大货计重⑦大货操作⑧操作装卸⑨周期性派费⑩集包费 − 政策返利），按 `网点×目的地区域/省份×重量段` 给费率；另有 `中转费`(区域×省×kg段+燃油费)、`出港派费`(始发省份首续重)、`大货/装卸`等分表。同时**出港运单本身已带总部实际结算值**（应收金额/中转费运费/附加费/加收费/预收面单费）。

## 2.2 卡片流优化（围绕"每天便捷导入"目标）

## 2.3 价格计算优化

- **逐票定价引擎**：输入(`父客户/报价ID + 目的省市 + 结算重量 + 付费方式`) → 输出(`基础运费 + 加收 + 面单费 = 应收`)，纯函数化、可单测（memory 已记"Express 计费缓存单测方式"）。
- **命中率护栏**：出港运单存在 `报价ID 缺失/数据处理提示≠正常计费` 的行，定价前先报"命中率/异常率"，未命中的走"客户默认报价/区域均价"兜底并打标，绝不静默丢弃。
- **报价版本与生效日期**：矩阵/加收/一口价都要带生效区间（测算表里加收已带"开始/结束日期"），按运单业务时间取当时报价，避免改价污染历史。
- **加收叠加与边界**：明确"省加收 + 市加收 + 周期性加收 + 流向考核加收"的叠加顺序与重量段边界(`30-999` 等)归属，统一用左闭右开。

## 2.4 成本计算优化（标准 vs 实际 双轨）

- **标准成本**：用 `太仓系统成本测算` 10 项费率，按运单逐票算（首续重/kg段/始发省/目的省）。
- **实际成本**：用出港运单已带的"总部实际结算值"+ 三家交易明细里的中转费/面单/派费/操作。
- **差异分析**：`标准−实际` 按 网点/目的省/重量段 出差异表，既校验报价是否覆盖成本，也发现总部多扣。
- **后置项处理**：政策返利、燃油费附加、考核加收是事后/按票合并的，要在月末按"返利明细/考核结果"回冲到对应运单或经营单元，不要漏。

## 2.5 价→本→毛利 闭环（关键架构建议）

建立一张**运单逐票经营事实表**（一条/运单）：
`收入(报价) − 标准成本(10项) = 标准毛利`；`收入 − 实际成本 = 实际毛利`；带 `经营单元/品牌/客户/业务员/目的省/重量段/期间` 维度。
**这张表同时是三件事的唯一数据源**：(1) 自动凭证的收入/成本计提；(2) 阿米巴出港段的收入/面单成本/毛利；(3) 客户/网点毛利分析。一处取数，避免"凭证、报表、分析"三套口径打架。

# 四、用卡片流替代钉钉 OA（自动获取费用报销/支出数据）

## 4.1 现状与差距

**费用数据现状（钉钉OA导出）**：两条流程——`对外付款(公司直付)` 423 笔/773万、`费用报销(垫付)` 997 笔/183万（"已分类"样本 1422 笔/957万，全部"审批通过"）；47 类费用，成本中心=费用归属单元(太仓总公司/浏河/市场部/城区/南郊/沙溪)；字段含 流程类型/报销明细/付款事由/金额/付款对象/收款账户/附件/审批编号/创建人(部门)/审批状态/审批记录/完成时间。

**本系统现有能力**：
- `STOTOP.Module.Workflow` 审批引擎 + `STOTOP.Module.OA` 模块；CardFlow 有 `Models/Approval` 审批模型——即"卡片走审批后再生成凭证"的链路有基础。
- 自动凭证：费用报销 41 条规则已就绪，可直接由卡片生成 `借 费用科目(辅:成本中心) / 贷 应付或银行`。

**差距**：缺面向业务的"报销表单设计 + 移动端发起 + 审批节点配置 + 附件/发票 + 出纳付款回填"的一体化体验（钉钉强在移动端发起与审批）。

## 4.2 两条路线

**路线 A（过渡，1-2 周）——先打通"自动获取费用数据 + 自动凭证"**
继续用钉钉发起审批，但把钉钉导出(或 API 拉取)的费用数据当作一类"卡片"导入：
- 新增"费用报销"卡片模板，字段映射钉钉schema；以 `审批编号` 幂等；
- 命中费用报销凭证规则(41+补 11 类) → 自动生成凭证，贷方按"对外付款=银行/垫付=其他应付款"；
- `费用归属单元` → 经营单元辅助核算，直接进阿米巴期间费用。
- 收益：立刻实现"无需手工记费用账 + 费用进阿米巴"，零自建审批。

**路线 B（目标，4-8 周）——自建报销卡片，彻底替代钉钉**
在本系统内建"报销/付款"卡片：表单(费用类别/金额/费用发生日期/费用归属单元/付款对象/收款账户/附件发票) → Workflow 审批流(提交人→部门负责人→财务→出纳) → 审批通过自动凭证 → 出纳在系统内执行/回填付款(`贷 1002 银行`) → 数据天然进阿米巴。
- 复用：CardFlow 审批模型 + Workflow 引擎 + 41 条凭证规则 + 经营单元辅助核算。
- 需建：报销表单设计器/固定表单、移动端发起页、审批节点配置、附件与发票管理、出纳付款回填、与凭证联动。
