# STOTOP 首次上线与快速发布管线（阶段 0）—— 设计

- 日期：2026-06-20
- 状态：已脑暴定稿，待用户复审
- 决策性质：战略方向 + 阶段 0 具体设计

---

## 1. 背景与核心决策

### 1.1 用户诉求
转入快速迭代模式：**做完一个功能就上线给用户用，根据反馈再迭代**。为此用户最初设想「根据现有成果重启一个项目，从框架开始重新构建，方便分模块开发、快速合并测试上线」。

### 1.2 调研结论（关键事实）
- STOTOP 已经是一套**成熟的模块化单体**：后端 **22 个独立业务模块**（独立 `.csproj`，在 WebAPI 分别注册装配）、**约 6.4 万行 C#** / 1970 文件、**660** 个前端 Vue/TS 文件、**104** 个测试文件、30 天内 497 次提交。
- 用户**早已在按「一个功能一个 spec→plan→实现」的节奏**开发（近期 specs 印证）。
- **已有一份完整的 IIS 发布设计**（`2026-06-09-iis-release-versioning-design.md`），但**仅是纸面 spec，仓库内没有任何 publish/deploy 实现脚本**；**无 CI**（无 `.github/workflows`）、**无 Docker**。
- 经确认：唯一核心痛点是 **「上线又慢又怕」**，且 **「还没正式上线」**。

### 1.3 核心决策：不重写框架
理由：
1. **重写是目标的反面。** 目标是「最快上线 + 拿反馈」，而从框架重建意味着数月内无法上线任何东西，先把已能运行的 6.4 万行重堆一遍——与「快速迭代」直接对冲。
2. **重写会把已填平的坑全部重开。** 记忆库本身是一份「踩过并修好的雷」清单（CardFlow 自动凭证三层匹配 fallthrough、EF 全局 NoTracking 身份冲突、辅助核算 `IgnoreQueryFilters` 被组织过滤滤掉、脱敏链明文泄露等），均为多会话对抗式验证换来。
3. **「快速迭代」是发布/反馈管线问题，不是代码架构问题。** 开发这端已是迭代式、模块化；缺的纯粹是「功能做完之后，怎么安全、快速、可回滚地送到用户面前」这后半截。

**真正要做的不是重启项目，而是在现有成果之上补一条快速、安全、可回滚的发布闭环，并把首发范围砍到一个核心切片。**

### 1.4 底座决策：继续 IIS/Windows，缓上容器
- 当前第一优先级是「把线上跑起来」；此时每引入一个新未知数（Docker、镜像仓库、容器宿主、编排）都是让上线卡住的新路。
- .NET 10 容器化技术上不难，但其**运维周边**（Windows/Linux 容器选择、SQL Server 连通、Hangfire 后台任务、SignalR、上传文件共享存储、加密的 `db-connections.json`）是另一个完整项目。
- 已有完整 IIS 发布设计，离上线最近的路是**把它实现出来**而非更换底座。
- 容器的价值在「水平扩展 / 多机一致性 / 多语言服务」——当前是一应用、一库、一人、零上线，YAGNI。撑不住单台 IIS 时再迁移一条**已能用**的管线。

## 2. 目标与非目标

### 目标（阶段 0）
- 打通 **首次上线 + 一键可重复、可回滚、带安全网的部署**。
- 首发**只上一个切片：「申通数据→经营报表」完整管线**（三类导入→计费/自动凭证/派件量→阿米巴经营报表），其余 21 模块入口收口。
- 让「上线一个功能」= 「合并 → 跑部署脚本 → 几分钟上线，几秒回滚」。

### 非目标
- ❌ 不重写框架、不重构架构。
- ❌ 不上容器 / 云原生（阶段 2 视需要再评估）。
- ❌ 不一次性上线 22 个模块。
- ❌ 不做多机水平扩展。
- ❌ 不做真·按网点灰度放量（阶段 2 再说；阶段 0 只用 feature-flag 让半成品默认关）。

## 3. 首发切片：「申通数据 → 经营报表」完整管线（含阿米巴）

### 3.1 用户面价值
**头部价值面向经营层：阿米巴经营报表（方案B，业务方向一等公民的损益）。** 用户上传申通三类 Excel（全部走 CardFlow 上传），系统自动算计费、自动生成凭证、聚合派件量，最终在阿米巴报表呈现按业务方向/网点/经营单元的经营损益。财务侧的「上传→自动凭证→复核/过账」是这条管线的中段产物。

这条管线被真实数据验证得最透（记忆：申通总部交易明细导入实测跑通、120 凭证借贷平衡、映射正确；阿米巴批次 1-6 + 派件量接入均已并入 master），测试也最厚。

### 3.2 范围边界（依赖闭包已代码核实，2026-06-20）
阿米巴报表 `AmoebaPLService` **直读三张事实表**（裸 SQL，不调用上游模块的 C# 服务）：`EXP出港运单_计费结果`（billing 收入/出港件量）、凭证分录（voucher 损益）、`STG申通派件日明细`（deliver 派件量），外加 depreciation/estimate/手工填报。三张事实表由三个 CardFlow 导入喂数：

| 导入源（均 CardFlow 上传） | 处理 | 事实表 |
|---|---|---|
| 申通出港运单 | Express 计费引擎当 CardFlow 插件算（规则「出港运单价格计算」） | `EXP出港运单_计费结果` |
| 申通总部交易明细 | 自动凭证引擎 | 凭证分录 |
| 申通派件结算 | 直接入暂存 | `STG申通派件日明细` |

- **开放（IN，运行时闭包）**：System（登录/组织/权限）、CardFlow（上传/暂存/插件规则/自动凭证引擎/导入校验工作台）、Finance（凭证/科目/账套/阿米巴报表/分摊引擎）、**Express 的计费/定价引擎**（`PricingPlugin`/`CostPlugin`/`BillingService`/`BillingBulkWriter`，作为 CardFlow 插件被调用）。
- **关闭（GATED OUT）**：CRM、Quality、HR、Salary、Dormitory、Vehicle、Conference、Contract、Insurance、KSF、Points、PPV、Supplier、Task 等——代码留库，入口关闭。**Express 的控制器/菜单/UI 也关闭**——阿米巴只需其引擎产出的事实表，不需暴露 Express 前台。
- ⚠️ **仅剩待核实**：CardFlow 是否运行时依赖 Workflow（记忆：Workflow 仍提供事件/派发等底层协作）——实现期确认，决定 Workflow 是开放最小必要部分还是可全关。

### 3.3 入口收口机制
对未发布模块做到「用户点不到、也调不到」，候选手段（实现期定具体组合）：
- 不注册其控制器 / 不暴露其路由；
- 不种入菜单（前端动态路由不放行）；
- feature-flag / 配置开关默认关闭；
原则：**开放最小必要入口集，其余默认关闭**。注意 Express 是「引擎开、前台关」的特例：计费引擎随 CardFlow 插件链路工作，但其控制器/菜单收口。

### 3.4 降级阀（billing 收入列）
首发目标是**完整管线含 billing 收入**。但设一道降级阀：若就绪门禁（§5①）发现 **出港运单计费路径**比凭证链脆（测试/真实数据验证不足），则 **billing 收入列暂缓上线**（标「待接入」），其余阿米巴（voucher+deliver+折旧+估算+手工）照常首发——**不让复杂的三层计费拖住整个首发**。

## 4. 发布管线架构（阶段 0）

落地形态：**一个 PowerShell 部署脚本**（建议放 `scripts/release/`），实现 `2026-06-09-iis-release-versioning-design.md` 的核心。步骤为流水线，**任一步失败即中止，绝不半上线**：

```
preflight（build + 全测试通过 = 门禁；不过直接停）
  → 后端 dotnet publish -c Release + 前端 vite build
  → 打版本包 STOTOP-vX.Y.Z+<sha>.zip（含 release-manifest.json）
  → 【安全网】部署前 SQL Server 全库备份
  → 复制到不可变版本目录（如 D:\STOTOP\releases\vX.Y.Z）
  → 【门禁】跑 EF 迁移（只进不退）；失败 → 中止，不切站点
  → 切换 IIS 站点物理路径到新版本目录
  → 【门禁】/health 健康校验；不过 → 自动回滚到上一版
  → 保留最近 5 版 + 追加 deploy-history.jsonl
```

- **回滚命令**：读 `deploy-history.jsonl`，把 IIS 物理路径指回上一成功版本（秒级；只回应用，不回数据库）。
- **共享数据**：敏感配置（`db-connections.json`）、uploads、logs 放 shared 目录，不入版本包。
- **版本驱动**：Git tag 驱动版本号（沿用既有 IIS spec 约定）。

## 5. 四道安全网（逐条对应四项顾虑）

### ① 代码质量/技术债 → 「首发就绪门禁」
范围砍小后，只需为**首发切片**背书，而非全部 6.4 万行；用户碰不到的模块债不阻塞上线。最怕的 CardFlow 凭证引擎恰是被锤得最狠、测试最厚的部分。
- **机制**：首发前对切片做一次集中体检——切片相关**全测试跑绿** + **一次全特性整体审**（经验：逐任务审会漏集成缝，须整体审）+ preflight。
- **重点对象：Express 三层计费引擎**（出港运单计费路径）。它是切片里最复杂、验证最不足的一段，也是 §3.4 降级阀的判定依据——就绪门禁须对它做真实数据验证，给出「可随首发」或「billing 列降级」的明确结论。

### ② 上线范围太大 → 「首发切片 + 入口收口」
见 §3。首发只开 CardFlow 凭证链，其余入口关闭、代码留库。

### ③ 数据库迁移风险 → 备份 + 门禁 + 预演 + 只进不退
现在还没上线、生产库基本是空的/种子数据，**第一次迁移风险极低，正是立纪律的最佳时机**。
- 每次部署**先自动全库备份**（真正的安全网：应用回滚 + 库恢复 = 可救）。
- **迁移只进不退、绝不改已发布迁移**（已知铁律：MigrationRunner 不可删迁移）。
- 迁移是**带门禁的步骤：失败立即中止、不切站点**。
- **真库副本先预演迁移**（沿用 IIS spec 两阶段晋级：测试服先跑）。
- 处理好已知 FK 顺序坑（记忆：Finance V9 被 FK_FIN 阻塞、CardFlow V23 版本号撞车）。

### ④ 上线后没人兜底 → 「系统自己喊疼 + 几秒可恢复」
- **`/health` 探活 + 钉钉告警**：复用现有钉钉集成（Daily/WeeklyReportBotJob），挂探活 ping + 未捕获异常告警，近零成本实现「出事先告诉你」。
- **一键回滚**（§4）+ **部署前全库备份**（③）= 应用与数据都能救。
- **小范围首发**（切片小 + 用户少）直接压低事故数量。
- 诚实边界：不保证零事故，但保证**可见得快、恢复得快、影响面小**——这是一人值守扛得住的前提。

## 6. 服务器选型与初始化（因服务器需新搭）

作为阶段 0 的一个任务，给出选型考量与默认推荐，待用户拍板：
- **考量清单**：成本；内网部署还是公网暴露（安全）；快递数据是否含 PII / 数据合规；能否连到现有 SQL Server；运维便利（一人值守）。
- **默认推荐**（待确认）：国内云厂商一台 Windows Server 主机（便于挂域名/证书/外网访问 + 快照备份），SQL Server 可同机起步、后续分离。若数据合规要求内网，则用公司内网一台 Windows 机器。
- **初始化清单**：Windows + IIS + .NET 10 Hosting Bundle + SQL Server 连通 + shared 目录（配置/uploads/logs）+ HTTPS 证书 + 防火墙/端口 + 备份目录与保留策略。

## 7. 阶段路线图

- **阶段 0（现在做）**：服务器初始化 → 最小部署脚本（build→包→切站点→health）→ 数据库安全（备份+迁移门禁）→ 首发切片定义与入口收口 → 切片就绪门禁 → 可观测与告警 → 首次上线 → **回滚演练（用之前先证明能恢复）**。
- **阶段 1（水到渠成）**：把 build+测试门禁搬进 CI（GitHub Actions 自托管 Windows runner），合并 master 自动构建+测试，部署仍一键；配合短命特性分支（trunk-based）。
- **阶段 2（真有需要才做）**：按网点灰度放量的真 feature-flag；撑不住单机再评估容器/多机。

## 8. 风险与未决问题
1. **出港运单计费路径就绪度（最大风险）**：Express 三层计费引擎是切片里最复杂、验证最不足的一段。就绪门禁须真实数据验证；不达标则走 §3.4 降级阀（billing 列暂缓），不拖整体首发。
2. **CardFlow 是否运行时依赖 Workflow**：需代码核实（记忆：Workflow 仍提供事件/派发等底层协作），决定 Workflow 开放最小必要部分还是可全关——否则入口收口可能漏关/误关致 500。其余依赖闭包（Express 引擎、Finance、三导入源）已于 2026-06-20 核实。
3. **服务器选型**需用户拍板：成本 / 内外网 / 数据合规三者权衡。
4. **钉钉告警**：已发现现成 `AlertBotJob.cs`（WebAPI/Jobs），复用其通道挂探活+异常告警的可行性需核实（与定时报表 Bot 是否同 webhook）。
5. **迁移预演环境**：单环境起步时，真库副本预演的具体做法（恢复一份生产备份到测试库再跑迁移）。
6. **入口收口的具体手段组合**（控制器/路由/菜单/flag）需在实现期统一拍定；注意 Express「引擎开、前台关」特例。

## 9. 验收标准（阶段 0「done」的定义）
- [ ] 能用**一条命令**把整条「申通数据→经营报表」管线部署到真实服务器，`/health` 绿。
- [ ] 能登录、上传申通三类 Excel、跑出借贷平衡的凭证；**未发布模块入口（含 Express 前台）确实点不到、调不到**。
- [ ] **阿米巴经营报表能渲染出按业务方向/网点的损益**（billing 收入随首发或按 §3.4 降级阀明确标注「待接入」）。
- [ ] 部署**前自动全库备份**已发生且可见。
- [ ] 迁移失败能**自动中止、不切站点**（演练过一次失败路径）。
- [ ] 能**一键回滚**到上一版，且**演练过一次**。
- [ ] 服务异常时**钉钉收到告警**（演练过一次）。
- [ ] 首发切片相关**全测试绿** + 完成一次整体审。
