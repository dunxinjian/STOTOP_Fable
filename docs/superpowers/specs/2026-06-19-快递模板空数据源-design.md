# 快递行业模板空数据源修复（P1-4）— 设计

- 日期：2026-06-19
- 范围：为「快递行业标准模板」(FIN科目模板 FID3) 补明细科目数据，修复建账选快递模板必报错。
- 优先级：P1（前端建账选快递模板即报错，开箱不可用）

## 1. 问题与事实依据

「快递行业标准模板」(FID3) 只建了模板头、无任何明细科目，导致两条建账初始化路径都报错：

- **路径 A（前端建账实际走）**：建账表单选模板 `formData.templateId`（`web/src/views/finance/AccountSetManage.vue:335`）→ `createAccountSet({templateId})` → 后端 `CreateAsync` 在 `TemplateId>0` 时调 `ApplyTemplateAsync`（`src/STOTOP.Module.Finance/Services/AccountSetService.cs:202-204`）→ 读 `FIN科目模板_明细` 为空 → 抛「模板中没有科目数据」（`AccountTemplateService.cs:288`）。
- **路径 B（潜在死路，前端不触发）**：`InitializeAccountSetAsync` 把 `express-delivery` 映射成 `sourceId=-2`（`AccountSetService.cs:353`）→ `GetSourceAccountsAsync` 查 `F账套ID=-2` 无科目 → 抛「模板数据源中未找到科目数据（源ID: -2）」（`:392`）。

事实核实：
- `FIN科目模板_明细`（实体 `FinAccountTemplateItem`，表名见 `FinAccountTemplateItemConfiguration.cs:11`）在 FinanceSeeder/迁移中**零 INSERT、零 CREATE**。
- 账套2（太仓美申）有 427 个科目种子；账套1（石家庄申通）482 个。
- `ApplyTemplateAsync`（`AccountTemplateService.cs:274-341`）按 `FLevel` 排序父先建，用 `idMapping[模板项FID → 新科目FID]` 还原父子；模板项 `F父ID` 引用**父模板项的 FID**。
- 1002（银行存款）/1012（其他货币资金）下各有大量账套私有的银行账户子科目：实测 1002 下 100201~100241（含三级 10020901~10020904）、1012 下 101201~101232，合计约 70+ 个。这些是各账套真实银行账户，**不应进行业标准模板**。

## 2. 修复设计

### 2.1 数据源决策
**快递行业标准模板(FID3) 的明细 = 账套2（太仓美申）的最新科目快照**，但排除 1002/1012 下的银行账户子科目（账套私有数据）。

### 2.2 新增 MigrateV11（FinanceSeeder）
为 FID3 补 `FIN科目模板_明细`，幂等可重跑。SQL 主体：

```sql
-- 幂等：先清空 FID3 现有明细
DELETE FROM [FIN科目模板_明细] WHERE [F模板ID] = 3;

-- 若 FID 为自增列，需 IDENTITY_INSERT ON（参照科目种子写法）
SET IDENTITY_INSERT [FIN科目模板_明细] ON;

INSERT INTO [FIN科目模板_明细]
  ([FID],[F模板ID],[F编码],[F名称],[F类别],[F余额方向],[F级次],[F父ID],[F是否末级],[F辅助核算],[F外币],[F计算单位],[F排序])
SELECT [FID], 3, [F编码], [F名称], [F类别], [F余额方向], [F级次], [F父ID],
       CASE WHEN [F编码] IN (N'1002', N'1012') THEN 1 ELSE [F是否末级] END,
       [F辅助核算], [F外币], [F计算单位], 0
FROM [FIN科目]
WHERE [F账套ID] = 2
  AND NOT ([F编码] LIKE N'1002%' AND LEN([F编码]) > 4)
  AND NOT ([F编码] LIKE N'1012%' AND LEN([F编码]) > 4);

SET IDENTITY_INSERT [FIN科目模板_明细] OFF;
```

要点：
- **模板项 FID 复用账套2科目 FID、F父ID 直接复制** → 父子层级天然保持（`ApplyTemplate` 的 idMapping 正确还原）。
- **排除 1002/1012 后代**（编码以 1002/1012 开头且长度>4，含三级），**1002/1012 本身 `F是否末级` 强制 1** 变叶子。
- **引用完整性**：被排除的只有 1002/1012 后代，保留科目的父链不受影响。
- **时序**：V11 必须在 V9/V10（账套2科目重建/调整）之后，确保账套2科目已就位。
- **表存在性**：`FIN科目模板_明细` 由 EF 实体配置管理；迁移前需确认表已建，否则兜底 `IF NOT EXISTS ... CREATE TABLE`（实现时勘察确认建表机制）。

### 2.3 路径 B 口径统一
`AccountSetService.cs:353` 把 `express-delivery => sourceId = -2` 改为 `=> sourceId = 2`，让 industry 腿也指向账套2，与模板数据源口径一致、消除潜在死路。

### 2.4 模板说明更新
FID3 模板说明由「源自石家庄申通2025账套快照」更新为「源自太仓美申账套快照（排除各账套私有银行账户子科目）」，与实际数据源一致。

## 3. 测试

> 数据迁移类修复，可测性弱，诚实说明分层：

- **ApplyTemplate 集成测试**（InMemory + 真 Repository）：seed FID3 模板头 + 若干明细项（含一个父项+子项、一个 1002 叶子项）→ `ApplyTemplateAsync(3, 新账套)` → 断言科目按数量建出、父子映射正确、1002 作为叶子无子。证明「FID3 有明细即可建账」机制成立。
- **迁移 SQL 验证**（无法 InMemory 单测——不跑原生 SQL）：靠真库跑迁移后人工/脚本验证：① `SELECT COUNT(*) FROM [FIN科目模板_明细] WHERE [F模板ID]=3` == 账套2科目数 − 账套2的 1002/1012 后代数；② `FIN科目模板_明细` 中 1002/1012 的 `F是否末级`=1、无其后代；③ 走 `ApplyTemplate(3)`/建账选快递模板不再抛错。
- `sourceId -2→2` 改动靠编译 + 既有回归。

## 4. 验证方式

- `dotnet build` Finance 0 错误。
- `dotnet test tests/STOTOP.Module.Finance.Tests --arch x64`：ApplyTemplate 集成测试红→绿，既有不回归。
- 真库跑迁移后按 §3 验证 FID3 明细行数与建账可用。

## 5. 非目标（YAGNI）

- 不统一两套初始化实现（`ApplyTemplateAsync` vs `InitializeAccountSetAsync` 的内联复制）。
- 不补 FID1/FID2 模板明细，只修快递 FID3。
- 不处理已建账套的补科目（路径 B 的 `initializeAccountSet` 兼容分支本就能跑通）。
- 1002/1012 之外的科目原样复制（不做其它裁剪）。
