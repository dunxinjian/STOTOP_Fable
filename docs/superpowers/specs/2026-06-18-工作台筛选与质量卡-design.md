# 工作台首页：合并来源筛选 + 统一质量卡 — 设计

- **日期**：2026-06-18
- **范围**：工作台首页 `/workhub` 的两处信息架构收敛（源自 DESIGN-IS-2026-06-18 审计的第 1、2 点）
- **裁决依据**：`DESIGN-IS-2026-06-18/03-verdict.md`（总分 18/30，#4 理解性、#10 克制各 1 分）
- **不在范围**：第 3 点（空态密度）、第 4 点（导航面收敛）、第 5 点（卡片整体改真按钮的可达性）。本次仅在改 chip 时顺带把 chip 写成 `<button aria-pressed>`。

## 背景与问题

审计发现工作台首页两处结构性冗余：

1. **同维度双筛选控件**：中栏顶部 6 个统计 chip（标签 审批/异常/任务/运单/合同/积分）与下方「来源」复选框组（标签 OA/质量/任务/运单/合同/积分）是**同一筛选维度**（相同 6 个 key：oa/quality/task/datacenter/contract/points），却是**两套控件 + 两套标签**。
   - 证据：`web/src/views/workhub/WorkHubCenter.vue:55-62`（statsItems）vs `:64-71`（sourceOptions），渲染 `:509-522` vs `:551-560`。
   - 附带 bug：chip 高亮由本地 `activeSource` ref 维护（`:51`、`:258-266`），与 `hub.filters.value.sources` 是两套状态，`onSourcesChange` 又把 `activeSource` 清空（`:268-272`），二者会 desync。

2. **两个质量部件看似矛盾**：中栏「质量预警」与右栏「运单与仓储质量」其实是**两个不同子系统**，只因都呈现「质量 · 待处理 N」且相邻而显得自相矛盾。
   - 中栏 `QualityAlertBar` → `getQualityDashboardStats` → `/quality/dashboard/stats`（**通用异常框架**），入口 `/quality/exceptions`。证据 `QualityAlertBar.vue:31-42`、`api/quality.ts:4-6`。
   - 右栏 `QualitySummaryCard` → `getWorkHubQualitySummary` → `/workhub/quality-summary`（**数据质量中心**：待处理店铺/空店铺行/未识别网点），入口 `/express/quality-center/dashboard`。证据 `QualitySummaryCard.vue:22-36`、`api/qualityCenter.ts:333-335`。
   - 两组件各自 5 分钟轮询（`QualityAlertBar.vue:49`、`QualitySummaryCard.vue:42`），口径与刷新各自为政。

## 关键约束（取数层已查证）

- 后端来源筛选**支持多选**：`useWorkHub.ts:161-163` 的 `buildParams()` 以 `filters.sources.join(',')` 传 `sources`。故合并后保留多选不需后端改动。
- `hub.setFilter('sources', value)` 会触发 `fetchItems(true)+fetchStats()`（`useWorkHub.ts:266-270`）。
- 两个质量接口均已存在且可前端并行调用，无需新增后端端点。

## 设计 A：合并来源筛选

**目标**：来源筛选只有一个控件、一套标签、一套状态。

1. **chip 升格为唯一来源筛选，多选 toggle**
   - 点击 chip：在 `hub.filters.value.sources` 中增删对应 key，然后调用 `hub.setFilter('sources', nextSources)`（复用其内置的 fetchItems+fetchStats）。
   - chip 的 `active` 态**直接派生**自 `hub.filters.value.sources.includes(stat.key)`，不再用本地 `activeSource`。
2. **删除「来源」复选框行**
   - 删除模板里的 `<a-checkbox-group :options="sourceOptions">`（`:554-560`）、`sourceOptions` 常量（`:64-71`）、`onSourcesChange`（`:268-272`）、`activeSource` ref（`:51`）、`toggleSourceFilter` 的整体替换逻辑（`:258-266`，改为多选增删）。
   - 筛选栏保留：优先级下拉、时间范围、重置、多选按钮。
3. **可达性顺带**：chip 由 `<div @click>` 改为 `<button type="button" :aria-pressed="active">`。
4. **标签口径单一**：仅保留 chip 一套标签（审批/异常/任务/运单/合同/积分）。

**边界/不变**：多选模式（`enterMultiSelect` 等）与批量操作逻辑不动；`重置` 仍清空含 sources（`resetFilters` 已含）。

## 设计 B：统一质量卡

**目标**：质量只有一处主入口，两个域一眼可辨；中栏只在真正紧急时出现。

1. **新增 `web/src/composables/useQualitySummary.ts` 作单一真源**
   - 并行调用 `getQualityDashboardStats()`（异常域）与 `getWorkHubQualitySummary()`（数据质量域）。
   - 暴露两域的 `{ pending, overdue }`（异常域 overdue 取 `overdueCount`，数据质量域取 `overdueWarning`），及派生 `hasOverdue = 异常.overdue>0 || 数据.overdue>0`。
   - **一处 5 分钟轮询 + onMounted 拉取**，组件卸载时清理。供「质量卡」与「中栏紧急条」共享，取代两组件各自的 `setInterval`。
2. **右栏 `QualitySummaryCard.vue` 重构为「质量」卡，按域两行**
   - 卡头：图标 + 「质量」。
   - 行 1「异常处理 / 通用异常框架」：`待处理 N · 超时 M`，点击 `router.push('/quality/exceptions')`。
   - 行 2「运单与仓储数据 / 数据质量中心」：`待处理 N · 超时 M`，点击 `router.push('/express/quality-center/dashboard')`。
   - 计数为 0 时该行数字灰显（`--text-3`）。
   - 数据改由 `useQualitySummary` 提供（移除组件内自有 fetch/轮询）。
3. **中栏 `QualityAlertBar.vue` 改为超时门控**
   - 显示条件由 `pendingCount > 0` 改为 `useQualitySummary().hasOverdue`（任一域 overdue>0）。
   - 文案按域列出超时数，例：`异常 2 项已超时 · 运单数据 1 项已超时`，右侧「查看 →」跳 `/quality/exceptions`（或就近指向超时所在域，实现期取较简单的固定跳 `/quality/exceptions`）。
   - 数据改由 `useQualitySummary` 提供。

**已确认的行为变化**：当存在「待处理但未超时」的质量项（如截图 待处理1/超时0），中栏紧急条**不再出现**；该项仅在右栏质量卡「异常处理」行计数，并照常作为 source=异常 工作项进入 Feed。信息不丢，仅不再抢占中栏顶部。此收敛符合「中栏只留真正紧急告警」的决策。

## 测试与验证

- 无前端测试运行器（见记忆「前端 type-check 基线红」），以 `vite build` 兜底编译，不依赖 `vue-tsc -b` 当门禁。
- 手动验证（preview 工作流）：
  1. chip 多选：连点两个来源 → Feed 与计数按多源过滤；再点取消其一 → 过滤更新；`重置` 清空。
  2. chip 高亮与实际 `sources` 一致（验证 desync bug 已消除）。
  3. 右栏质量卡两行计数/入口正确；0 时灰显。
  4. 构造 overdue>0 → 中栏紧急条出现并分域显示；overdue=0、pending>0 → 中栏紧急条隐藏、右栏「异常处理」仍计数。
  5. 仅一处轮询（网络面板确认不再有重复的两组质量请求）。

## 风险

- `getQualityDashboardStats` 返回结构为 `{totalExceptions,pendingCount,overdueCount,todayNewCount}`（`QualityAlertBar.vue:17-22`），`getWorkHubQualitySummary` 为 `{pendingTotal,todayNew,overdueWarning}`（`qualityCenter.ts:311-315`）；composable 需做字段归一，避免命名混用。
- 紧急条隐藏「待处理未超时」项属预期内行为变化，需在 PR 描述中点明，避免被误读为漏显。
