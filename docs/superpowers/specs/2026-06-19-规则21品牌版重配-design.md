## 1. 背景与目标

品牌版科目已落库，但库 live 凭证规则 FID=21《Tai申通总部交易明细生成凭证规则》仍指向旧网点版科目。实证（直连测试库 47.105.65.51/stotop dump）：

- **88 个 accountId 全部失效**（dead 88 / valid 0），全指旧网点版 200xxx 段。
- **85 个 accountCode 中 60 个在品牌版账套2不存在**；剩 25 个"存在"的也**绝大多数是同码异义**。

→ 申通总部交易明细的自动凭证生成现已不可用。

**目标**：把规则21 重配到品牌版科目恢复凭证生成；并借此修复随之暴露的既有缺陷（漏记、静默丢弃、匹配键冲突）。

**范围（已定）= 全面**：85组重指 + 补 gap B9/C15 + 修 createDraft + 配齐辅助核算 + 解匹配键冲突。

## 2. 现状事实底座（库 live 实证）

规则21 配置（`CF自动插件_规则` FID=21，列 `F规则配置JSON`）：

| 项 | 值 |
|---|---|
| mode / version | rulesBased / 2 |
| accountSetId | 2（太仓/美申） |
| 规则组 | **85 组** |
| matchingLayers | exactMatchField=`F科目编码`、categoryField=`F费用名称`、summaryField=`F业务摘要` |
| keyFields | [F科目编码, F费用名称, F业务摘要] |
| unmatchedAction | `createDraft` |

**凭证结构**：每组 2 行 = 贷方(收入/成本，固定科目) + 借方(动态按 `F网点编号` 匹配 4 网点 → 4 个总部应付科目)。共 **170 分录行**、340 matchRules（仅 4 个 distinct 总部应付 FID 200916–919）。

**两个既有缺陷（非品牌版引入）**：
1. **unmatchedAction=createDraft 名不副实**：`AutoVoucherHandler` 只对 `error` 分支特殊处理（[AutoVoucherHandler.cs:222](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherHandler.cs:222)），`createDraft`/`skip` 走同一路径——未匹配行被记录(`FUnmatchedRows`/`FUnmatchedDetailsJson`，可重试)但**不生成草稿凭证**。
2. **匹配键冲突**：11 个费用编码被 2–3 组占用且二三层（exactCategories/categoryKeywords/summaryKeywords）**全为空**，引擎只能按 order 首组抢占、其余永不命中（如 `2202.02.50` → 按需服务/按需派送考核激励/按需派费 三组）。根因是来源 xlsx 映射本身二义（同费用编码+同费用名却对多个旧科目）。

## 3. 引擎契约与可复用基础设施

- **科目解析**（[AutoVoucherHandler.cs:599 `ResolveAccountIdV2`](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherHandler.cs:599)）：① 固定 `AccountId` → ② 动态 `AccountMatchRules[].AccountId` → ③ 兜底 `DefaultAccountId`。**只认 accountId，从不用 accountCode 反查**。
- **跨账套重映射基础设施**（[AccountMappingResolver.cs](src/STOTOP.Module.CardFlow/Services/Import/AccountMappingResolver.cs)）：`RemapRuleConfigAsync` 遍历规则JSON所有科目位（含动态行/兜底/辅助核算），**按 accountCode 在目标账套反查 FID 并覆盖 accountId（双写）**，反查不到的进 `MissingCodes`；另有 `LoadCodeToFidMapAsync`/`ResolveByCodeAsync`。
- ⚠️ **不能机械套用 RemapRuleConfigAsync**：它假设 code 不变，而本场景旧码↔新码存在同码异义（如旧 `50010301` 基础派费 ≠ 品牌版 `50010301` 退件操作费）。
- **落地策略**：重配工具先按业务确定品牌版目标 **code** → 用 `LoadCodeToFidMapAsync(账套2)` 反查填 **FID**（双写 code+id）。**不硬编码 600/700xxx 数字**。

## 4. 方案（7 项决策已定）

| # | 决策点 | 选定 |
|---|---|---|
| 1 | 范围 | 全面（重指+补gap+修静默漏记+配辅助核算） |
| 2 | 落地 | 库 live UPDATE + 种子追平 |
| 3 | unmatchedAction | 修复 createDraft 真生草稿 |
| 4 | 借方总部应付 | 单一 220201 + 品牌辅助 |
| 5 | business_direction | 按费用编码固定每组 |
| 6 | 收入侧无精确叶 | 逐项财务定（v3 已按铁律落收入侧） |
| 7 | 匹配键冲突 | 补判别维度（业务提供关键字） |

### 4.1 目标科目重指（85 组核心）
**收入/成本硬判据 = 费用编码前缀**（关键铁律）：`2202.*`（取 F发生额收入）→ 必落收入 `500*`（贷）；`1122.*`（取 F发生额支出）→ 必落成本 `5401*`（借）；**严禁交叉**。实证：41 组 2202.* / 44 组 1122.*，干净二分。
按品牌版 13 性质组归并（思路依据 design 科目梳理 §5.1；**目标 code/name/fid 取自实测库 brandcoa，design 文档二级编码已偏移、不用**）。完整映射见附件草案（v3）：**54 高置信**、0 编造 fid、**0 前缀交叉**（独立脚本 + 对抗审查双重校验，109 条全过）、全部命中账套2实存叶子。

### 4.2 借方总部应付改造
借方 → 单一固定 `220201 总部应付`(fid 700125) + `express_brand`=fixed 申通；**删掉 4 条网点 matchRules**。网点维度由贷方损益科目的 `outlet` 辅助核算承载（符合品牌版"往来按品牌、网点走辅助"设计）。

### 4.3 辅助核算配齐 4 维
贷方损益科目：`outlet`=dynamic(`F网点编号`) / `express_brand`=fixed 申通 / `business_direction`=fixed(逐组判定 OUT/IN/CMB，分布 OUT21/IN20/CMB44) / `business_unit`=fixed(申通经营单元，**取值待确认**)。

### 4.4 补 gap 24 组
- **B** `2202.*` 结算收入 9 种 → 品牌版 5001 收入叶；**C** `1122.*` 费用 15 种 → 5401 成本叶。均缺费用名明细，**先占位 + createDraft 待补录**，逐条须财务补费用名后定科目（金额大者优先：#86 122万/#87 92万/#88 19.5万）。
- **A** 资金类 3 种（0401充值取款/0301取款调账/0201充值）= 资金往来非损益，**不入规则21**；如需自动化另立资金类规则（后续）。

### 4.5 unmatchedAction 修复
`createDraft` 改为真正为未匹配行生成「待补录草稿凭证」（科目待填/状态=草稿），修复点在 [AutoVoucherHandler.cs:222](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherHandler.cs:222) 附近新增 `createDraft` 分支 + 凭证生成支持"无科目草稿"。⚠️ 需确认草稿凭证形态是否破坏借贷平衡校验。

### 4.6 匹配键冲突 → 补判别维度
11 个冲突费用编码，由业务提供 `F业务摘要`/费用名关键字，给冲突组配 `categoryKeywords`/`summaryKeywords` 二三层区分，使同费用编码多组能分别命中。冲突清单见审查报告 §四。

### 4.7 收入侧无精确叶 → 逐项财务定
按费用编码铁律，2202.* 收入组一律落收入侧（已纠正 v2「收入错塞成本」的系统性错误）。但品牌版收入侧缺 8 类专属叶（环保袋/集包/进港增值/航空提货/网点服务费受款/中转/全网出港等，order 4/5/6/8/9/10/40/41），v3 暂落收入侧最贴近叶 `50010302 其他操作费` + 标低置信。候选——**甲：维持 50010302 / 乙：品牌版新增对应收入叶后改记**——财务逐项定。另成本侧政策类 3 组（#75/#76/#77）同理无专属成本叶，暂落 54010404/54010613。清单见映射草案 §1.2 / §2.6。
