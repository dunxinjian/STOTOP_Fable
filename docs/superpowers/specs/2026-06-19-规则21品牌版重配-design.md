## 1. 背景与目标

品牌版科目已落库，但库 live 凭证规则 FID=21《Tai申通总部交易明细生成凭证规则》仍指旧网点版科目。实证（直连测试库 47.105.65.51/stotop dump）：**88 个 accountId 全失效、85 个 accountCode 中 60 个在品牌版不存在** → 申通自动凭证生成空窗。

**目标**：把规则21 重配到品牌版科目恢复凭证生成；并修复随之暴露的既有缺陷（漏记、静默丢弃）。

**范围 = 全面**：基于真实数据重建映射 + 补全覆盖 + 修 createDraft + 配齐辅助核算 + 配套 4 项科目表调整。

## 2. 现状事实底座

### 2.1 库 live 旧规则21（将被取代）
`CF自动插件_规则` FID=21：mode=rulesBased v2、accountSetId=2、85 组、unmatchedAction=createDraft、matchingLayers={exactMatchField:F科目编码, categoryField:F费用名称, summaryField:F业务摘要}。每组 2 行=贷方损益固定科目 + 借方动态按 `F网点编号` 匹配 4 网点总部应付。88 accountId 全失效。

### 2.2 数据源真实组合（映射的 ground truth）
统计 `Taicang/申通系统交易明细/` **全部 93 文件（88 有效 + 4 个 0 字节空文件，含 88 个 .xls + 5 个 .xlsx）**，得 **104 个 `(费用编码, 费用名称)` 组合，一码一名**（每个费用编码唯一对应一个费用名）。

数据源关键列：`费用科目编码`(一级) / `费用名称`(二级) / `业务摘要`(三级=日期+费用名) / **`费用科目收付类型`**（总部应付/总部应收，直接定收入/成本）/ `发生额(收入)`/`发生额(支出)`/`网点编号`。

> **重要修正**：之前误以为有「匹配键冲突」（同编码多组），实为旧 85 组源于过时 xlsx 的冗余拆分。真实数据一码一名，**冲突不存在**。

### 2.3 既有缺陷
`unmatchedAction=createDraft` 名不副实：[AutoVoucherHandler.cs:222](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherHandler.cs:222) 只对 `error` 分支处理，`createDraft`/`skip` 走同一路径——未匹配行被记录(`FUnmatchedRows`/`FUnmatchedDetailsJson`，可重试)但**不生成草稿凭证**。

## 3. 引擎机制（已核实，支持「编码+名称」组合）

- **三层匹配**（[AutoVoucherMatchingEngineV2.cs](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherMatchingEngineV2.cs)）是**分层候选 + fallthrough**，非「编码 AND 名称」联合键：Layer1 用 `F科目编码` 查 `_exactCodeIndex`（`code→groupId`，**同编码只映射第一组**）→ Layer2 `F费用名称` → Layer3 `F业务摘要`，前层不中才降级。
- **组内「编码+名称」组合**靠分录行的 `ConditionField`/`ConditionValues`（[AssignWithinDirection](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherMatchingEngineV2.cs:408)）：一个费用编码归一组，组内多分录行按 `F费用名称` 条件分流到不同科目。
- **科目解析**（[AutoVoucherHandler.cs:599 `ResolveAccountIdV2`](src/STOTOP.Module.CardFlow/Services/Handlers/AutoVoucherHandler.cs:599)）：固定 `AccountId` → 动态 `AccountMatchRules` → 兜底 `DefaultAccountId`，**只认 accountId**。
- **跨账套重映射**（[AccountMappingResolver.cs](src/STOTOP.Module.CardFlow/Services/Import/AccountMappingResolver.cs)）`LoadCodeToFidMapAsync`/`ResolveByCodeAsync`：按 code 反查 FID。重配工具据此填 FID，**不硬编码数字**。
- **结论**：申通真实数据一码一名 → **每个费用编码归一组、一组一目标科目**，无需组内条件分录、无冲突；引擎现成支持。

### 4.1 目标科目映射（已逐条核对，v6）
基于真实 104 组合 → 品牌版账套2 损益叶，已**逐条人工核对**（56 条改动）。结果：损益 101（收入侧 25 / 成本主营 70 / 期间费用 6）+ 资金类 3 剔除。独立脚本校验 **0 跨侧 / 0 fid 问题**。完整对照见映射草案附件。

**收付类型铁律**：总部应付(2202.\*)→收入 5001/5051(贷)；总部应收(1122.\*)→成本费用(借)；资金类(0201/0301/0401)→不进规则21。
**例外**：`#菜鸟仓配项目付费`→`220301 预收面单`（预收负债，非损益，业务实质为预收面单款）。

**核对确立的业务规则**：派费**成本侧=出港OUT、收入侧=进港IN**；考核/赔款按所属业务逐项判出/进港；总部大客户项目=替总部大客户做操作、总部付操作费→操作费收入。

### 4.2 成本费用三分
1122.\* 内部按性质分：主营结算（派费/中转/操作/集包/面单/考核罚款/赔款/物料/服务费）→ `5401`；行政资金类（网点租金→560204/水电→560205/利息→560301/通讯→56020901/手续费→560302）→ `56xx`；税费 → `5403`（本批无）。

### 4.3 借方 + 辅助核算
借方一律=单一 `220201 总部应付`(fid700125)+`express_brand`=fixed 申通。贷方损益科目辅助核算四维：`outlet`=dynamic(`F网点编号`)/`express_brand`=fixed 申通/`business_direction`=fixed(逐组核对值 OUT/IN/CMB)/`business_unit`=fixed(申通经营单元，取值待确认)。

每个费用编码 → 一个规则组（`ExactCodes`=[费用编码]）+ 贷方损益科目 + 借方 220201。一码一名，无需组内条件分录。matchingLayers 保留（categoryField/summaryField 备未来一码多名场景）。
