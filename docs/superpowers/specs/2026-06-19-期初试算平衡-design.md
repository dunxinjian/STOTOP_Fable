# 期初试算平衡校验（P1-3）— 设计

- 日期：2026-06-19
- 范围：期初余额录入增加「借方合计 = 贷方合计」试算平衡校验，后端为权威、前端即时反馈并拦截保存。
- 优先级：P1（可保存不平期初污染整个账套后续滚动）

## 1. 问题与事实依据

期初录入当前**不做任何借贷平衡校验**，可保存「借方 100 万、贷方 80 万」的不平期初，后续所有期间期末余额据此逐期滚动，试算平衡表从第一期起就不平且极难溯源。

- **前端**：期初表底部 summary 只用两个独立 `reduce` 并排展示借方合计、贷方合计（`web/src/views/finance/AccountManage.vue:170-184`），不算差额、不比较、不拦截保存（`handleSaveInitialBalances` 直接提交，`:786-804`）。
- **后端**：`SaveInitialBalancesAsync`（`src/STOTOP.Module.Finance/Services/AccountService.cs:366-429`）已「不信任客户端」地校验了科目归属本账套 + 仅末级可录，唯独漏了借贷总额平衡校验，即 upsert 落库。
- **关联缺陷**：`AccountController.SaveInitialBalances`（`src/STOTOP.Module.Finance/Controllers/AccountController.cs:161-167`）**没有** try/catch（不像 Create/Delete），后端抛的 `InvalidOperationException` 会以未处理异常变 500，前端只能看到通用「服务器内部错误」。本次加了平衡校验抛异常，必须同时补异常转译，否则前端拿不到「期初不平」的具体原因。

> 同代码库其它写入路径（结账 `AccountPeriodService`、凭证、Excel 导入）都有平衡校验，唯独期初这条漏做，坐实是疏漏。

## 2. 修复设计

### 2.1 校验范围语义：全量快照（不信任客户端）
不只校验本次提交的 items，而是校验「提交覆盖后，该账套全部末级科目的期初」整体平衡：每个末级科目的期初借/贷取「本次提交值，否则库中现值（`PeriodId=0`），否则 0」，再比较 `Σ借` 与 `Σ贷`。与后端既有「归属/仅末级」校验同风格，前端漏发部分行也不会绕过。

### 2.2 后端纯函数校验器
新建 `src/STOTOP.Module.Finance/Services/InitialBalanceValidator.cs`（static class，纯函数便于单测）：

- `const decimal Tolerance = 0.005m`（金额 2 位小数容差）。
- `ComputeDifference(IEnumerable<long> leafAccountIds, IReadOnlyDictionary<long,(decimal Debit, decimal Credit)> existing, IReadOnlyDictionary<long,(decimal Debit, decimal Credit)> submitted) → decimal`：对每个末级科目取 `submitted` 命中值，否则 `existing` 命中值，否则 `(0,0)`，累加返回 `Σ借 − Σ贷`。
- `IsBalanced(decimal difference) → Math.Abs(difference) < Tolerance`。

### 2.3 接入 SaveInitialBalancesAsync
在现有「归属 / 仅末级」校验之后、upsert 循环之前：
1. 末级科目集合 = `accountIds − parentIds`（现有变量已具备）。
2. 查该账套 `PeriodId=0` 的全部 `FinAccountBalance`，建 `existing` 字典（`FAccountId → (FBeginDebit, FBeginCredit)`）。
3. items 建 `submitted` 字典（`AccountId → (DebitBalance, CreditBalance)`）。
4. `diff = InitialBalanceValidator.ComputeDifference(末级集合, existing, submitted)`。
5. 若 `!IsBalanced(diff)`，抛 `InvalidOperationException`，消息含借方合计、贷方合计、差额（2 位小数），例如：`$"期初借贷不平衡，借方合计 {totalDebit:F2}，贷方合计 {totalCredit:F2}，差额 {diff:F2}"`。

> 校验通过后才执行原有 upsert，逻辑不变。

### 2.4 控制器异常转译
`AccountController.SaveInitialBalances` 包 `try/catch (InvalidOperationException ex)` → `ApiResult.Fail(ex.Message)`，与 `Create`/`Delete` 一致。

### 2.5 前端即时反馈 + 拦截（AccountManage.vue）
- 新增 computed（替换模板内现有 inline `reduce`，DRY）：
  - `initialDebitTotal` / `initialCreditTotal`：对 `initialBalances.filter(isLeaf)` 求和。
  - `initialDifference = initialDebitTotal − initialCreditTotal`。
  - `isInitialBalanced = Math.abs(initialDifference) < 0.005`。
- summary：合计行下增「借贷差额」展示，差额 ≠ 0 时红字高亮（差额为 0 时正常色）。
- 保存按钮（`:121`）`:disabled="savingInitial || !isInitialBalanced"`。全 0 未录时差额 0、视为平衡、允许保存（允许清空/留空）。

## 3. 测试

- **后端纯函数** `InitialBalanceValidatorTests`：
  - 借贷相等 → IsBalanced true；
  - 借 > 贷 / 贷 > 借 → false 且 ComputeDifference 符号正确；
  - 容差边界（差 0.004 → 平；差 0.006 → 不平）；
  - 提交值覆盖库中现值（同一末级科目 existing 与 submitted 都有，取 submitted）；
  - 末级科目未提交时用库中现值参与合计；
  - 全空 → 差额 0、平衡。
- **后端集成**（InMemory + 真 Repository，参照 `AmoebaPLServiceTests` 构造模式）：`SaveInitialBalancesAsync` 对不平 items 抛 `InvalidOperationException`、对平衡 items 正常落库——证明校验真接进保存流程。
- **前端**：项目无前端测试运行器、type-check 基线本就红，靠 `vite build`（或 type-check 过滤本文件）兜底 + preview 手动验证：录不平 → 差额红字 + 保存禁用；调平 → 差额 0 + 保存可用。

## 5. 非目标（YAGNI）

- 损益类科目禁录期初（另一条 P2，单独处理）。
- 数量期初 / 外币期初（核算闭环，另立）。
- 「结束初始化 / 账套启用」状态机闸门（P1 另一条，更大，单列 spec）。
- 已保存的历史不平期初不自动订正（需运维介入）。
