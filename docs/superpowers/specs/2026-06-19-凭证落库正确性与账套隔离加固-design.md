# 凭证落库正确性与账套隔离加固（P0 · A+B+C）

- **日期**：2026-06-19
- **范围**：`src/STOTOP.Module.Finance`（VoucherService / VoucherController / Filters）+ `web/src/views/finance/VoucherEntry.vue` 前端配套
- **来源**：2026-06-19 凭证模块全面审计（89 条已复核发现）中的 P0 致命项
- **本 spec 含**：簇 A（账期/账套落库 + 结账校验）、簇 B（写入事务原子化）、簇 C（越权 IDOR + 补权限属性）
- **本 spec 不含**：簇 D（外币三字段持久化，单列）；所有 P1/P2/P3

---

## 1. 背景与问题

手工凭证录入主链路（`VoucherEntry.vue` → `VoucherController` → `VoucherService`）存在四类动摇账务正确性与数据隔离的根基性缺陷。本 spec 处理其中三簇（A/B/C）：

1. **账期/账套落 0**：新增凭证 `FPeriodId` 永远为 0、`FAccountSetId` 落 0，导致按账期查询查不到、月度凭证号不重置、断号检查漏检、结账保护被绕过、列表选中账套后看不到自己刚建的凭证。
2. **写入无事务**：凭证头与分录逐条 `SaveChanges`，中途失败产生不平衡/无分录的脏凭证；`UpdateAsync` 先删后插无回滚，失败时永久破坏原凭证。
3. **跨账套/跨组织越权（IDOR）**：所有按 id 的读写操作只按裸 `FID` 取数，不校验归属组织/账套，改 URL 即可读/审/删他账套乃至他组织凭证；多个写/读端点完全缺权限属性。

### 设计原则
**后端权威**。不信任前端传入的 `periodId`/`accountSetId`，后端自行从「凭证日期 + 账套」推导并强校验；前端只做显示与取号预览的配套。落库逻辑收敛在 `VoucherService` 这一唯一出口。

## 2. 簇 A：账期/账套落库 + 结账校验

### A1. 抽共享方法 `ResolvePeriodAsync`
在 `VoucherService` 内新增私有方法（逻辑复用自 `CardFlowVoucherBridge.cs:42-56`）：

```csharp
private async Task<FinAccountPeriod> ResolvePeriodAsync(DateTime date, long accountSetId)
{
    var period = await _periodRepository.Query()
        .Where(p => p.FAccountSetId == accountSetId
            && p.FStartDate <= date
            && p.FEndDate >= date)
        .OrderByDescending(p => p.FStartDate)
        .FirstOrDefaultAsync();
    if (period == null)
        throw new InvalidOperationException(
            $"未找到账套 {accountSetId} 在 {date:yyyy-MM-dd} 对应的账期，请先建立该期间");
    return period;
}
```

- 查不到期间 → **抛错拒绝落库**（决策：缺账期一律拒绝，不自动建期间）。
- 返回完整 `FinAccountPeriod`，调用方据 `FIsClosed` 判断是否结账。

> 备注：`CardFlowVoucherBridge` 现状在调用 `CreateAsync` 前已自行解析 periodId 并传入；A2 让 `CreateAsync` 在 `PeriodId<=0` 时才解析，故 Bridge 传入的合法 periodId 不受影响，两者无冲突。Bridge 自身那段重复逻辑本 spec 不强制合并（保持改动聚焦）。

### A2. `CreateAsync` 落库修正
- **accountSetId**：控制器 `Create` 在 `accountSetId<=0` 时从请求头 `X-AccountSet-Id` 兜底解析后传入（权限过滤器 `RequireAccountSetPermissionAttribute.cs:43` 已在读此头）。在 service 内仍保留「`orgId==0 && accountSetId>0` 时按账套反查 orgId」的既有兜底。
- **periodId**：`request.PeriodId<=0` 时调 `ResolvePeriodAsync(request.Date, accountSetId)`，用返回的 `period.FID` 作为 `FPeriodId`；`request.PeriodId>0` 时沿用（兼容 Bridge）。
- **结账校验**：解析/传入的期间 `FIsClosed==1` → 抛 `InvalidOperationException("该期间已结账，不能新增凭证")`。不得依赖前端按钮禁用作为唯一防线。
- 取号 `GetNextNumberAsync` 使用解析后的真实 periodId。

### A3. `UpdateAsync` 落库修正
- 用 `ResolvePeriodAsync(request.Date, voucher.FAccountSetId)` 重新定位期间并写入 `voucher.FPeriodId`（现状 `voucher.FPeriodId = request.PeriodId` 会把它清零）。
- 结账校验顺序：
  1. 原 `voucher` 当前所属期间已结账 → 拒绝（保留现有 `FStatus==3` 与 period.FIsClosed 双重判断）。
  2. **新解析期间**若已结账 → 同样拒绝（防止把凭证移入已结账期）。
- **修复分录组织标记丢失**（独立发现）：重建分录时补写 `FOrgId`（取 `voucher.FOrgId`）与 `FDataScopeId`（取 `request.DataScopeId`），与 `CreateAsync` 对齐。

### A4. `SaveDraftAsync`
- 同样调 `ResolvePeriodAsync` 解析期间落库（草稿也不污染取号号段）。
- **不做结账硬拒绝**（草稿是提交前态）；但仍要求期间存在（查不到抛错）。

### A5. 复制/冲销日期与期间一致
- `CopyAsync` / `ReverseAsync` 新单 `FDate = DateTime.Today`，并用 `ResolvePeriodAsync(DateTime.Today, source.FAccountSetId)` 解析 `FPeriodId`（现状沿用 `source.FPeriodId` → 跨月错配）。
- 目标（今天的）期间若已结账 → 拒绝（红冲/复制不得落入已结账期）。
- 效果：红字冲销自然落到当前开放期，符合实务。

### A6. 前端配套（`web/src/views/finance/VoucherEntry.vue`）
- `watch(() => form.value.date)` 时在 `periodList` 中按日期区间匹配并回写 `form.value.periodId`（用于「年/期」显示与取号预览正确）；匹配不到则置 null 并由 `isClosedPeriod`/保存时的后端错误兜底提示。
- `fetchNextNumber` 使用回写后的 `periodId`。
- 不改变「后端权威」：即便前端 periodId 仍为空，后端也能从日期解析。

## 3. 簇 B：写入事务原子化

对 `CreateAsync / UpdateAsync / ReverseAsync / CopyAsync / SaveDraftAsync` 用显式事务把「凭证头 + 全部分录」包成单一原子单元：

```csharp
await using var tx = await _context.Database.BeginTransactionAsync();
try
{
    // ...写凭证头 + 逐条写分录（Repository 各自 SaveChanges 共享同一 scoped DbContext，纳入同一事务）...
    // 操作日志写入也置于事务内
    await tx.CommitAsync();
}
catch
{
    await tx.RollbackAsync();
    throw;
}
```

- **前提确认**：`STOTOPDbContext` 与 `IRepository<>` 均为 Scoped（`Program.cs`），一次请求内共享同一 DbContext，故 Repository 内部的多次 `SaveChanges` 会落在同一显式事务中，可整体回滚。
- **操作日志** `OperationLogService.LogAsync` 放事务内（失败回滚则不留日志）。
- **事件派发** `VoucherPendingAuditEvent` 移到 **Commit 之后**（消费者只看已提交数据，避免读到将被回滚的数据）。
- `GetByIdAsync` 读回放在 Commit 之后。

> 与簇 A 的关系：A 引入的结账/缺账期校验在事务开启前完成（fail-fast，不必要时不占事务）。

## 4. 簇 C：越权 IDOR 加固 + 补权限属性

### C1. 服务层按 id 操作加组织/账套归属（核心）
当前 `GetByIdAsync / AuditAsync / UnAuditAsync / DeleteAsync / UpdateAsync / CopyAsync / ReverseAsync / CompleteRecordAsync / BatchAuditAsync` 均按裸 `FID` 取数。统一改造：

- 弃用绕过全局过滤器的 `_voucherRepository.GetByIdAsync(id)`（`FindAsync`），改为：
  ```csharp
  var voucher = await _voucherRepository.Query()
      .FirstOrDefaultAsync(v => v.FID == id && v.FOrgId == currentOrgId);
  ```
  → 堵死**跨组织**访问。
- 再校验同组织内账套归属，**分两种来源场景**：
  - **有 `X-AccountSet-Id` 请求头**（前端交互调用）：要求 `v.FAccountSetId == currentAccountSetId`，不等即越权拒绝。
  - **无请求头**（后台任务/自动凭证，无 HttpContext）：降级为**仅组织校验**（已由 `FOrgId == currentOrgId` 完成），不再强制账套相等，避免误拒自动凭证。
- 新增私有 `GetCurrentAccountSetId()`：读取 `X-AccountSet-Id` 头（与过滤器同源），无头时返回 0 表示「不约束账套」。
- 组织/账套不匹配统一返回「凭证不存在」语义（不泄露存在性）。`BatchAuditAsync` 中越权项按「跳过」计入失败明细。

### C2. 控制器补缺失的 `[RequireAccountSetPermission]`
（`src/STOTOP.Module.Finance/Controllers/VoucherController.cs`）

| 端点 | 当前 | 补充权限 |
|---|---|---|
| `Copy` (copy/{id}) | 无 | `VoucherCreate` |
| `Reverse` (reverse/{id}) | 无 | `VoucherCreate` |
| `SaveDraft` (draft) | 无 | `VoucherCreate` |
| `CompleteRecord` ({id}/complete-record) | 无 | `VoucherCreate` |
| `ReorderNumbers` (reorder/{periodId}) | 无 | `VoucherEdit` |
| `GetDrafts` (drafts) | 无 | `VoucherView` |
| `GetNextNumber` (next-number) | 无 | `VoucherView` |
| `GetPendingAuditCount` (pending-count) | 无 | `VoucherView` |
| `CheckGap` (check-gap) | 无 | `VoucherView` |
| `ExportTemplate` (export-template) | 无 | `VoucherView` |

> 权限属性是第一道（账套级权限）；C1 的服务层归属校验是第二道（凭证级归属）。两道都需要——权限属性只验「用户对头里的账套有无权限」，不验「凭证是否属于该账套」。

### C3. 明确不含（属 P1，后续 spec）
- 审核「制单人 ≠ 审核人」职责分离、仅 `FStatus==1` 可审。
- 冲销前置校验（仅已审核可冲、不可重复冲销、回写原单已冲销标记）。

## 5. 验证策略（TDD 优先 + 真库回归）

### 单元测试（`dotnet test --arch x64`，EF Core InMemory/SQLite 驱动 service）
- `ResolvePeriodAsync`：命中返回 / 查不到抛错 / 已结账期返回供调用方判断。
- `CreateAsync`：①`FPeriodId`、`FAccountSetId` 正确落库；②缺账期拒绝；③已结账期拒绝；④`PeriodId>0` 时沿用（Bridge 兼容）。
- `UpdateAsync`：①按新日期重解析期间；②移入已结账期被拒；③重建分录带 `FOrgId/FDataScopeId`。
- `Copy/Reverse`：新单期间按今日解析；今日期间已结账被拒。
- **IDOR**：他组织 / 他账套的 voucher 上 `Audit/Delete/GetById/Update` 被拒（返回不存在语义）。
- **事务回滚**：构造分录写入抛错 → 断言无孤儿凭证头落库（用 SQLite 内存库可测；InMemory 不支持事务，需 SQLite）。

> 测试隔离注意：service 重度依赖 `STOTOPDbContext` 与 `IHttpContextAccessor`。`IHttpContextAccessor` 用 mock 注入 `CurrentOrgId`/Claims/请求头；DbContext 用 SQLite in-memory（支持事务，能验回滚）。

### 真库回归（测试库手工）
建单 → 列表按账期可见 → 编辑改日期跨月 → 审核 → 跨月冲销落当前开放期 → 已结账期建单被拒。比对 `FPeriodId/FAccountSetId/FOrgId` 落库值与凭证号按期重置。

## 6. 验收标准

1. 手工新增凭证 `FPeriodId`、`FAccountSetId`、`FOrgId` 均正确落库；列表选中账套后可见。
2. 凭证号按真实期间分组、月度从 1 重置；断号检查覆盖这些凭证。
3. 已结账期间无法新增/编辑/移入凭证（后端拒绝，非仅前端禁用）。
4. 复制/冲销新单落在当前开放期（按今日解析），跨月不再错配。
5. 任一分录写入失败时不产生孤儿/不平衡凭证头（事务回滚）；`UpdateAsync` 失败不破坏原凭证。
6. 跨组织、跨账套的按 id 读写一律被拒；列出的 10 个端点补齐权限属性。
7. `UpdateAsync` 重建的分录带 `FOrgId/FDataScopeId`。
8. 全部单测通过（`--arch x64`）；真库回归链路通过。

## 7. 风险与回归面

- **Bridge 兼容**：`CardFlowVoucherBridge` 走 `CreateAsync(request, creator, accountSetId)` 且传入合法 periodId，A2 改动对其无副作用（需回归一次 CardFlow 自动凭证生成）。
- **事务与现有「AddAsync 立即提交」约定**：本 spec 用显式事务包裹后，多次 SaveChanges 在同一事务内，不再各自即时持久化——需确认无依赖「逐条已提交」的旁路逻辑。
- **请求头依赖**：若某调用方未注入 `X-AccountSet-Id`（如后台任务无 HttpContext），C1 的账套校验需对「无头」场景降级为仅组织校验，避免误拒自动凭证。
