## 1. 背景与目标

申通总部业务系统导出的一批网点质控数据（太仓城区公司 320288），反映网点员工在工作中的异常、不达标、质量问题。目标是把这些数据**定期导入本系统 → 整理成统一模型 → 形成质量看板**，进而（二期）**推送给责任部门与员工去确因、督促改善，并据此对部门/人员做奖惩**。

数据样本路径：
- 明细：`Taicang\网点质控数据\申通网点\申通数据明细`（22 个文件）
- 汇总：`Taicang\网点质控数据\申通网点\申通数据展示页`（9 个文件）
- 同级还有 `极兔网点`、`韵达网点` 目录 —— **后续要接入其它承运商**，承运商是模型的一等维度。

## 2. 范围与边界

**一期（本 spec 实现目标）**
1. 多文件批量导入：一次拖入整批文件，按表头内容自动路由到对应数据源。
2. 全部 9 大质量域入库（数据先齐），落地到 `STG_申通_*` 暂存表。
3. 归一：把异构明细/汇总整理成统一质控模型（4 张 `QL` 表）。
4. 承运商质控看板：网点总览 / 员工质量 / 问题件追踪 三视角，细到员工（工号）。

**二期（本 spec 仅预留接口、不实现）**
- 系统自算 KPI：用 `QL检测规则` + `DetectionService` 从指标/事件算出网点/员工得分。
- 奖惩：基于得分计算奖惩，落 `QL绩效记录`；考核金额据实可对接财务 540106/500106。
- 推送确因督促：把命中规则的问题提升为 `QL异常单`，经 `DispatchService` 派发（钉钉 + 待确认的其它渠道）→ 状态机闭环（待处理→处理中→超时→已关闭）。

**奖惩取向（用户已定）**：系统自算 KPI（按规则从明细自行计算网点/员工奖惩，不依赖总部金额）；粒度细到员工/小件员级，网点级作为自然上卷层并承载只能到网点的指标。

**待确认项**：二期推送除「钉钉」外用户勾选了「其它渠道」，具体渠道未明确，二期开工前确认。

## 3. 数据源分析

### 3.1 两层结构
- **明细层**：一行一票（运单号为主键），是「证据」。
- **汇总层（展示页）**：一行一网点/天，是总部算好的「记分牌」，多数带 `预估/预计考核金额`。

两层是同一指标的「明细 ↔ 汇总」关系。

### 3.2 九大质量域
1. 物流信息质量（及时 / 完整 / 准确）
2. 揽收时效
3. 出仓时效
4. 交货滞留
5. 末端派送 / 签收时效
6. 积压与遗失
7. 投诉与赔付（含真实赔款金额）
8. 虚假签收与履约质检
9. 拦截与渗透建站

### 3.3 贯穿关联键（整理的基础）
- **运单号** —— 原子键；一票可在多个域出现，串成「问题件全景」，多域命中即重点件。
- **网点编码（320288）** —— 组织键，对应 `ExpNetworkPoint.FCode`。
- **员工工号（如 3202880036）** —— 责任键，对应 `ExpSalesman.FEmployeeNo`（工号前 6 位即网点码）；定位到人的关键。
- **统计日期** —— 时间键。
- **电商平台**（拼多多/抖音/淘系/京东/菜鸟…） —— 渠道维度。

姓名脏（如 `城区屈梦幻400`、`城区029申亚楠17638932823`，带编号/电话）：**以工号作准，姓名原文仅展示**。工号缺失的行（如完整性明细的签收员有空值）归到网点级，`F员工工号` 留空。

## 4. 总体架构

```
31 个 Excel（9 域，明细+汇总两层）
        │  多文件拖入 + 表头内容路由
        ▼
STG_申通_*  暂存表（约 27 张，沿用 CardFlow 框架）
        │  归一作业（CardFlow 后置插件 / 定时作业）
        ▼
统一质控模型（并入 Quality 模块，QL 前缀 4 表）
  · QL承运商质量事件   （事实：一票×域）
  · QL员工日质量指标   （工号×日，预聚合）
  · QL网点日质量指标   （网点×日，预聚合）
  · QL质量问题字典     （配置：问题类型→域/严重度/是否考核/是否可归人）
        │
        ├── 承运商质控看板（前端 quality 视图，三视角）        ← 一期
        └── QL检测规则 + DetectionService → QL异常单
              → DispatchService（钉钉/任务/消息）→ QL绩效记录  ← 二期
```

复用既有能力，不重造：
- **导入**：CardFlow 暂存导入框架（配规则 + 建 STG 表，上传页/工作台数据源无关）。详见 [[stotop-cardflow-staging-import]]。
- **质量框架**：`STOTOP.Module.Quality` 是一套通用质量管理框架（异常单/检测规则/派发/绩效/复盘/看板），其中 `DetectionService`、`DispatchService` 当前是 `// TODO` 空壳，正是二期要落地的部件。
- **看板范式**：前端 `web/src/views/quality/dashboard/QualityDashboard.vue`（KPI 卡 + ECharts + 明细表 + 组织切换自动取数）。
- **组织/员工/网点建模**：`IOrgScoped` 全局过滤、`ExpNetworkPoint`、`ExpSalesman`。
- **推送基建（二期）**：Task 模块 `TmNotification`（站内信）+ `DingTalkMessageService`（钉钉工作通知/待办）。

## 5. 接入层设计（导入）

### 5.1 多文件拖入 + 内容路由
不靠文件名（这批文件名很乱），靠**表头列指纹**匹配：每个数据源的 `CfFlowDefinition.FMatchPattern.columnIdentifier` 配一组「必含列」。批量上传时逐文件读表头 → 必含列全命中即路由到该流程 → 命中不了的文件进「待认领」提示用户手工指派。

需两处小增强（其余零改动）：
1. 上传页支持多选 / 拖文件夹，逐文件路由（现状若仅单文件，加一层循环）。
2. 解析按内容判别格式：这批里 `抖音照片质检.xls`、`应拦截量数据报表.xls`、`虚假签收明细.xls`、`网点数据报表.xls` 实为 `.xlsx`（PK/ZIP 头），按扩展名走 xlrd 会失败 —— 解析前嗅探文件头（PK → openpyxl）。

## 6. 统一质控模型（归一）

并入 `STOTOP.Module.Quality`，表名走大写 `QL` + 中文（对齐 `QL异常单`/`QL检测规则`），实体类沿用 `Ql` PascalCase 命名（如 `QlCarrierQualityEvent`）。全部实现 `IOrgScoped`。

> 字段为依据真实数据列出的代表性集合，实现计划阶段最终敲定。

### 6.1 `QL承运商质量事件`（事实表，一行 = 一票 × 一个问题）
系统字段：`FID`、`FOrgId`、`F创建时间`、`F来源STG表`、`F来源行ID`、`F来源批次ID`
业务字段：`F承运商`、`F业务日期`、`F统计年月`、`F运单号`、`F网点编码`、`F网点名称`、`F员工工号`(可空)、`F员工姓名原文`(可空)、`F电商平台`(可空)、`F质量域`、`F问题类型编码`(→`QL质量问题字典`)、`F问题类型名称`、`F严重度`(0低/1中/2高/3紧急，对齐 `ExceptionPriority`)、`F是否考核件`、`F考核金额`(可空)、`F责任方`(本网点/外部网点/总部/客户)、`F关键字段JSON`(原始关键字段快照)
二期联动：`F是否已提升异常单`、`F关联异常单ID`(→`QL异常单.FID`)
- 业务主键去重键：`F承运商 + F运单号 + F质量域 + F问题类型编码 + F业务日期`
- 索引：`(F承运商,F业务日期,F网点编码)`、`(F员工工号,F业务日期)`、`(F运单号)`、`(F质量域,F问题类型编码)`、`(F来源批次ID)`
- 语义：同一票被多个域命中即多行，「多域命中重点件」一查即出；是问题件追踪与二期算分的原子层。

### 6.2 `QL员工日质量指标`（工号 × 日，预聚合）
键：`F承运商 + F业务日期 + F网点编码 + F员工工号`（唯一）
指标（对齐数据）：`F派件量`、`F当日派签量`、`F当日派签率`、`F应上门量`、`F未上门量`、`F按需上门率`、`F客诉发起量`、`F工单定责量`、`F客诉发起率`、`F虚假签收数`、`F照片质检不合格数`、`F派送超时T0~T3数`、`F揽收不及时数`、`F上传不及时数`、`F问题件数`、`F违规_虚假电联`、`F违规_无效电联`、`F违规_双签`、`F违规_照片定位虚假`、`F违规_签收文本不规范`、`F违规_引导代收`、`F回访真实率`、`F考核金额合计`
- 看板「员工质量」数据源 + 二期 KPI 自算输入。`小件员履约指标` 文件几乎是这张表的成品来源。

### 6.3 `QL网点日质量指标`（网点 × 日，预聚合）
键：`F承运商 + F业务日期 + F网点编码`（唯一）；含 `F网点名称`、`F片区`、`F省区`
分域指标（含考核金额）：
- 物流信息：`F揽收/派件/签收上传不及时率`、`F揽收/派件/到件缺失率`、`F不准确率`、`F到件不准确率`
- 揽收：`F及时揽收率`、`F未及时揽收量`
- 出仓：`F一频次出仓及时率`、`F未及时出仓量`、`F出仓预估考核金额`
- 交货滞留：`F滞留率`、`F考核滞留量`、`F滞留预估考核金额`
- 派送签收：`F一/二阶段及时签收率`、`F当天及时签收率`、`F派送预估考核金额`、`F有偿派费金额`、`F预计返款金额`
- 签收率考核：`F48h签收率`、`F签收率考核金额`
- 积压遗失：`F日均出港量`、`F日均进港量`、`F积压倍数`、`F超3/5/7天积压量`、`F遗失率ppm`、`F遗失量`
- 投诉：`F进港投诉量`、`F进港投诉率`、`F虚签投诉率`、`F7日虚签投诉量`
- 拦截：`F应拦截量`、`F拦截成功率`、`F及时转出率`
- 渗透建站：`F自建渗透率`、`F渗透率目标`、`F建站待完成`、`F喵柜激活格口数`
- `F考核金额合计`
- 承载「只能到网点、到不了人」的域；展示页 9 个汇总是现成来源。

### 6.4 `QL质量问题字典`（配置表，扩展点）
`FID`、`FOrgId`、`F承运商`(空=通用)、`F质量域`、`F来源问题类型原文`、`F问题类型编码`、`F问题类型名称`、`F问题大类`、`F问题小类`、`F默认严重度`、`F是否考核`、`F是否可归责到人`、`F默认整改时限小时`、`F状态`、`F备注`
- **加新域、调严重度、改是否考核，只动这张字典**，不改代码。

### 6.5 归一作业
- 形式：CardFlow 后置插件（批次导入完成后触发）或定时作业，二选一在实现计划定（倾向后置插件，跟随批次、可回滚）。
- 逻辑：每张 STG 表配一份映射（域、目标表[事件/员工指标/网点指标]、字段映射、问题类型字典查找）。
  - 问题清单类 STG（投诉/虚签/遗失/未到件/超时/照片质检不合格…）→ 生成 `QL承运商质量事件`。
  - 员工级汇总（小件员履约指标）→ upsert `QL员工日质量指标`；可聚合的明细也回填员工/网点指标。
  - 网点级汇总（展示页 9 张）→ upsert `QL网点日质量指标`。
- 承运商由来源 STG 表确定（`STG_申通_*` → 申通）。韵达/极兔接入：加各自 STG + 映射，落进同一套 `QL` 表（`F承运商` 区分），看板自动多一家可比。

## 7. 与 Quality 框架集成（二期落地，一期预留）

## 8. 质量看板（一期，三视角）

顶部统一筛选：**承运商（申通/…）+ 统计期/日期范围**。

- **视图 1 · 网点总览**（网点×期）：KPI 卡（签收率、出仓及时率、交货滞留率、积压量/倍数、遗失率ppm、虚签率、考核金额合计）+ 近 30 天多指标趋势 + 质量域问题分布 + 考核金额按域构成。源：`QL网点日质量指标` + 汇总 STG。

## 10. 风险与特殊处理

1. **坑文件三类**：
   - 投诉账单（137 列、双行分类表头）：按合并表头展开为单层列再映射；或只取所需列。
   - 签收率未达标考核（合并分段表头、大量 `Unnamed` 列）：按固定列位/分段解析。
   - 错命名 `.xls`（实为 xlsx）：解析前按文件头嗅探格式。
2. **多 sheet**：`物流信息指数` 含 3 个 sheet（及时/完整/准确），CardFlow 现仅解析首 sheet → 拆成 3 张汇总表分别导入，或给导入规则加 `sheetName`（小增强）。
3. **工号缺失**：归不到人的事件落网点级，`F员工工号` 空。
4. **脏名**：以工号为准，姓名原文仅展示。
5. **数据量**：未出仓/末端派送明细单期上千行、按期累积 → STG 建索引（批次/日期/网点/工号），指标表预聚合扛查询。
6. **多承运商**：一期只配申通，模型与看板从第一天带承运商维度，避免后期返工。

## 11. 一期验收标准

1. 一次拖入这批申通文件，能按表头内容自动路由、全部入对应 `STG_申通_*` 表；命中不了的文件给出明确「待认领」提示。
2. 重复导入同一期文件不产生重复数据（跨批次去重生效）。
3. 归一后 `QL承运商质量事件`/`QL员工日质量指标`/`QL网点日质量指标` 数据与源文件抽样核对一致（按 [[sample-dont-aggregate-business-data]] 采样核对）。
4. 看板三视角可用：网点总览 KPI 与趋势正确；员工质量能按工号排行并钻取到其质量事件；问题件追踪可多维筛选、能识别多域命中重点件、可导出。
5. 组织切换 / 承运商筛选 / 日期范围切换时数据正确刷新。
6. 全过程不触碰二期内容（不算 KPI、不派发、不写奖惩）。
