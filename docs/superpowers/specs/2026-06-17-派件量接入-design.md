# 派件量接入 — 设计 spec

> 拍板：用户 2026-06-17。范围：**A+B 都做**——派件量经 Excel 导入落 `STG申通派件日明细`（复用 CardFlow 暂存框架），阿米巴日/周报公共费分摊的 deliver 件量改从该表取数。

## 0. 背景与目标

阿米巴批次5-S4（日/周公共费按当月实际线性摊）的对抗式审查发现：公共费分摊的 **deliver 件量恒为 0**。根因——`发件/派件票量` indicator 是 `FIsManualEntry=1`（手工）节点，`MatchDataPointsToPLItems` 只匹配非手工项，故 `ComputeVolumeBasis` 读 `matched` 永远取不到派件量；send 有 billing 兜底，deliver 无源。后果：纯 deliver 基数公共费（进港操作工资）日/周恒摊 0、total 基数（房租/水电/折旧/摊销/财务费用）退化为仅 send 比例。

**真实数据源**（用户 2026-06-17 澄清）：派件量来自外部「每天每人派件明细」Excel（示例 `Taicang/每天每人进出港表-汇华.xlsx`：日粒度、按 结算日期×网点×业务员、列 `基础派费收费件量`=派件量；单文件一网点、每日导入 T-1/T-2、多网点）。

**目标**：派件量成为一等的、可按日期+网点裁切的 deliver 件量源（与 billing 出港件量平行），**根治 deliver=0 / scope 不感知 / 跨月周近似 三问题**。

## 1. 现状与依据（已核实）

- **既有 CardFlow 暂存导入框架**（配置驱动 ETL）：上传 `UploadCenter.vue`（源无关）→ `BatchTriggerService.MatchFlowDefinitionsAsync` 按列头/文件名自动匹配流程 → `CfBatch` → `FlowEngineService` → `ExcelInputPlugin` 按 `CF自动插件_规则.F规则配置JSON`（targetTable+columnMapping+keyFields+跨批次去重）通用解析 → `SqlBulkCopy` 落 STG 表。**无需自定义解析代码**。
- **STG 表两形态**：EF 实体型（`StgShentongOutbound`/`StgExpenseRecord`：实体+Config，强类型工作台/统计/撤销）vs 动态原始 SQL 型（`STG_出港运费`：纯 DDL）。本表列固定 → **用 EF 实体型**。
- **STG 建表陷阱（关键）**：`SchemaAutoSync` 与 `DatabaseSeederAdapter.CreateMissingTables` 都把 `STG` 前缀排除（各自 `ExcludedPrefixes` 含 `"STG"`）。**存量库不会自动建 STG 表**，必须在 `CardFlowSeeder` 追加显式 `IF NOT EXISTS … CREATE TABLE` 迁移（照搬 `MigrateV9` 建 `STG_出港运费`）。
- **导入工作台/上传页源无关**：`UploadCenter.vue`、导入验证工作台 `ImportCalculationValidationWorkbench.vue`（按 batchId）均零改动。唯一源下拉硬编码处是暂存浏览页 `StagingBrowser.vue`（仅 jt/st/yd）。
- **S2/S4 现状**：`ApplyCommonCostAllocation`（引擎 `Allocate(叶, 全额, scopeVol, fullVol)=全额×scopeVol/fullVol`）已就位；`ComputeVolumeBasis(points, matched, sendId, deliverId)` 现从 `matched` 取 deliver（恒 0，待改）。S4 已加跨月周/全期告警兜底（`e3ea2bc`）。

## 2. 总体架构

```
派件 Excel(每网点/每日) ──上传──> [CardFlow 通用导入] ──> STG申通派件日明细
                                                              │
                                                  AggregateDeliverVolume(起,止,组织,网点scope)
                                                              │  SUM(F基础派费收费件量)
                                                              ▼
  阿米巴日/周报 ── ApplyCommonCostAllocation ── ComputeVolumeBasis(deliver 由此传入)
                       (S4 当月基线 / S2 同期基线)
```

`STG申通派件日明细` 是 Part A（导入）与 Part B（取数）的解耦接口。

## 4. Part B — 报表取数（Finance / `AmoebaPLService`）

### B1. `AggregateDeliverVolume(startDate, endDate, orgId, scope)`

原始 SQL（参数化，平行 `AggregateBillingRevenue`）查 `STG申通派件日明细`：
```
SELECT ISNULL(SUM([F基础派费收费件量]),0)
FROM [STG申通派件日明细]
WHERE [F结算日期] >= @start AND [F结算日期] < @endExclusive   -- 半开，跨月周按周内实际天精确
  AND [FOrgId] = @org AND [FIsRevoked] = 0
  [AND [F网点编号] IN (@outlets…)]                            -- scope.Outlets 非空时
```
返回该范围+网点的派件量合计（long）。半开区间：`endExclusive = 末日.AddDays(1)`（与 billing/voucher 口径统一）。

### B2. `ComputeVolumeBasis` 改造

deliver 件量改由调用方传入（取自 `AggregateDeliverVolume`），send 仍 `billing SUM(WaybillCount)` → 发件票量 indicator 兜底：
```
ComputeVolumeBasis(points, matched, sendId, deliverCount)   // deliverCount 直接传入
  send = billingSend>0 ? billingSend : matched[sendId]?
  deliver = deliverCount
```
（既有 S2 单测随签名更新。）

### B3. `ApplyCommonCostAllocation` 接入

调用方先算两侧 deliver：
- **scoped deliver** = `AggregateDeliverVolume(本期起止, 组织, scope网点)`。
- **baseline deliver** = `AggregateDeliverVolume(基线起止, 组织, 全网点)`——S4 基线=所属当月、S2 基线=同期。
传入 `ApplyCommonCostAllocation` → `ComputeVolumeBasis`。**根治三问题**：deliver 有源（非 0）；按网点裁（scope 感知）；半开范围求和（跨月周按周内实际天精确，不再近似）。S4 的跨月周告警因此可移除或降级。

### B4. 派件票量 indicator 展示值改源

报表内 `派件票量` indicator 的显示值改从 `AggregateDeliverVolume(本期, 组织, scope)` 取（替代 manual 录入依赖）。发件票量类似可从 billing 取（既有）。**录入侧不再手工录派件量**。

### B5. 测试

- **A**：建表/导入规则冒烟（SQL Server，Skip 保护）；`ExcelInputPlugin` 解析派件示例落表（若可注入规则）。
- **B**：`AggregateDeliverVolume` 范围+网点+跨月周按天求和（需 DB，原始 SQL → SQL Server 冒烟或抽纯函数边界）；`ComputeVolumeBasis`/`ApplyCommonCostAllocation` deliver 传入后 total/deliver 基数公共费分摊正确（InMemory 内存点，复用 S2 测试样式）。

## 5. 数据流与口径

- **件量量纲可比**：send（billing 出港件量）与 deliver（派件结算件量）均为「件」，分子分母同量纲。total = send+deliver。
- **方向**：派件天然进港（IN）；`AggregateDeliverVolume` 不再按 business_direction 拆（表即进港）。scope 的 Direction=OUT 子报表，deliver 基数公共费按业务应为 0 或不适用——v1 按网点裁为主，方向维度暂不特殊处理（记风险）。
- **当月例外联动**：S4 当月基线经 `BuildPeriodDataPoints(当月,month)` 取公共费全额（含当月例外的估值），deliver 基线则经 `AggregateDeliverVolume(当月范围)` 取——两者口径独立但同期，比例自洽。

## 6. 范围边界 / 非目标

- 派费**金额**列只入库、**不接报表**（进港收入/成本仍走凭证）。
- 多维 scope 中派件量**主要按网点裁**；方向/项目维度 v1 不特殊处理。
- `StagingBrowser.vue` 「派件」tab 可缓做。
- 不为派件接 Pricing/Cost/AutoVoucher 等后续节点（纯导入落库）。

## 7. 风险与注意

1. **STG 前缀建表陷阱**：必须显式迁移 CREATE TABLE，否则存量库无表、导入即失败。
2. **列名逐字一致**：实体 Config / 迁移 DDL / 导入规则 columnMapping.dbColumn 三处列名必须完全一致；Excel 表头列名须与 columnMapping.excelColumn 逐字一致（示例表头多行重复 + `&` 等特殊字符，需核对真实表头，必要时用 aliases 兜）。
3. **跨批次去重键约束**：`ExcelInputPlugin` 跨批次去重字段最多取前两字段——去重键设计（网点+结算日期 或 业务员+结算日期）须与唯一索引、业务幂等期望一致。
4. **示例表头多行**：示例前 3 行疑似重复表头，`headerRow`/`dataStartRow` 须按真实文件定位。
5. **方向维度**：deliver 基数公共费在 Direction 子报表的口径未定，v1 仅按网点。
6. **AggregateDeliverVolume 原始 SQL**：与 billing 一样 InMemory 不可测，靠 SQL Server 冒烟 + 纯函数边界覆盖。
