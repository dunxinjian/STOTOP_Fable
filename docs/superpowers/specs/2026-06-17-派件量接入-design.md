# 派件量接入 — 设计 spec

> 拍板：用户 2026-06-17。范围：**A+B 都做**——派件量经 Excel 导入落 `STG申通派件日明细`（复用 CardFlow 暂存框架），阿米巴日/周报公共费分摊的 deliver 件量改从该表取数。

> **评审修订（2026-06-17，41-agent 评审 workflow，确认 24 项）**：本版按对抗式评审结论改写，根治"照搬模板"踩到的隐藏陷阱。三项设计抉择已拍默认值（可改）：
> - **去重键 → 方案 B**：4 字段唯一索引 + 关闭插件跨批次去重，重复导入靠唯一索引整批失败（语义干净、确定可行）。方案 A（合成列 `F去重键`+transformRules）依赖未验证的插件灌列能力，列为备选。
> - **展示改源 → v1 只做分摊根治**（§4.B2/B3）；indicator 展示改源 + 历史手工数据处置（原 §B4）延后，v1 不做（见 §6）。
> - **账套隔离 → 按组织**（与 billing `AggregateBillingRevenue` 对齐，派件物理量不随账套拆分）；`F账套ID` 仅入库、不参与去重/取数（见 §5）。

## 0. 背景与目标

阿米巴批次5-S4（日/周公共费按当月实际线性摊）的对抗式审查发现：公共费分摊的 **deliver 件量恒为 0**。根因——`发件/派件票量` indicator 是 `FIsManualEntry=1`（手工）节点，`MatchDataPointsToPLItems` 只匹配非手工项（`AmoebaPLService.cs:1092`），故 `ComputeVolumeBasis` 读 `matched` 永远取不到派件量；send 有 billing 兜底，deliver 无源。后果：纯 deliver 基数公共费（进港操作工资 FID 661）日/周恒摊 0、total 基数（房租/水电/折旧/摊销/管理/财务费用 FID 665-670）退化为仅 send 比例。

**真实数据源**（用户 2026-06-17 澄清）：派件量来自外部「每天每人派件明细」Excel（示例 `Taicang/每天每人进出港表-汇华.xlsx`：日粒度、按 结算日期×网点×业务员，列 `基础派费收费件量`=派件量；单文件一网点、每日导入 T-1/T-2、多网点）。**示例文件已解压核实**：sheet `data`（930 行），**前 3 行均为重复表头、数据从第 4 行起**（A4=`20260511`、B4=`150049`）→ 见 §3.A2 `dataStartRow=4`。

**目标**：派件量成为一等的、可按日期+网点裁切的 deliver 件量源（与 billing 出港件量平行），**根治 deliver=0 / scope 不感知 / 跨月周近似 三问题**。

## 1. 现状与依据（已核实，附 file:line）

- **既有 CardFlow 暂存导入框架**（配置驱动 ETL）：上传 `UploadCenter.vue`（源无关）→ `BatchTriggerService.MatchFlowDefinitionsAsync`（`BatchTriggerService.cs:488-596`，三轮：fullColumnIdentifier 精确→columnIdentifier 包含→fileNamePattern 回退）→ `CfBatch` → `FlowEngineService` → `ExcelInputPlugin` 按 `CfPluginRule.F规则配置JSON`（`ParseConfigAsync ExcelInputPlugin.cs:697-766`）通用解析 → `SqlBulkCopy`（`BulkCopyAsync :872-894`）落 STG 表。**无需自定义解析代码**。
- **STG 表两形态**：EF 实体型（`StgShentongOutbound`/`StgExpenseRecord`：实体+Config，强类型工作台/统计/撤销）vs 动态原始 SQL 型（`STG_出港运费`：纯 DDL）。本表列固定 → **用 EF 实体型**。
- **⚠ STG 建表陷阱（已核实）**：`SchemaAutoSync.cs:80` 与 `DatabaseSeederAdapter.cs:50` 的 `ExcludedPrefixes` 都含 `"STG"`。**存量库与新库都不会自动建 STG 表**，必须在迁移显式 `IF NOT EXISTS … CREATE TABLE`（照搬 `CardFlowSeeder.MigrateV9` 建 `STG_出港运费` 的写法 `:530-569`）。
- **⚠ 迁移版本（已核实）**：`MigrationRunner.ValidateSteps`（`:251,256-266`）强制版本从 1 严格 +1 连续递增、删步骤即抛异常（"不可删"=代码强制；"不可改" body=约定，存量库按已记录版本跳过不重跑）。**CardFlow 当前最高迁移版本 = V22**（`CardFlowSeeder.cs:44`）→ **新步骤 = V23**。
- **⚠ rule/流程种子位置（已核实，纠 §8 旧漂移）**：rule 3001/3002/3003 与流程首节点种子在 **`MigrateV4`**（`CardFlowSeeder.cs:324-384`，rule `:335/342/349`、流程节点 INSERT `:357-376`），**不在 MigrateV9**（V9 只建 `STG_出港运费` 表）。
- **⚠ 跨批次去重的隐式两字段约束（已核实）**：`ExcelInputPlugin` 跨批次去重物理上**只用前两字段**（`HashSet<(string,string)>` 二元组：`:586-597,1043-1055,1082-1083`），第 3+ 字段被静默丢弃、无校验无报错；配 3 字段更糟——临时表第 3 列恒 NULL 而 JOIN 用全字段（`:1065`）→ `NULL=值` 恒 false → **跨批次去重整段静默失效**。同批次内去重 `ComputeBusinessKey`（`:978-989`）用全部 keyFields 做 SHA256，不受两字段限制。**本 spec 据此选去重方案 B（见 §3.A1/A2）**。
- **⚠ crossBatchDedupFields 字段名口径（已核实）**：跨批次去重字段按 **DataTable 列名（DbColumn，带 F 前缀）** 校验 `dt.Columns.Contains`（`:471-478`），不含即静默禁用去重（仅 LogWarning）；而 `keyFields` 用 **Excel 中文名**（`ComputeBusinessKey` 按原始 row 字典取值 `:983`）——**两者命名域相反**。既有 rule 3002/3003 把 crossBatchDedupFields 写成无 F 前缀的 Excel 名（`:343/350`），意味着现网出港/费用导入的跨批次去重**一直处于禁用状态**（且 `STG_出港运费` 无唯一索引 → 等于完全不去重）。**照搬此结构会搬来该隐藏 bug，本 spec 不照搬（见 §3.A2）。**
- **⚠ ExcelInput 不写 FDataScopeId（已核实）**：纯导入路径 `CreateDataTable`/`BulkCopyAsync`（`:851-868,882-891`）不写 `FDataScopeId`/`FSourceWorkItemId`（恒 NULL）；`FDataScopeId` 仅凭证生成节点写 `FinVoucher`（`VoucherMigrationPlugin.cs:224/239`）。**故 `RevokeByDataScopeAsync` 对纯导入派件行恒 no-op（见 §3.A3）。**
- **导入工作台/上传页源无关**：`UploadCenter.vue`、导入验证工作台 `ImportCalculationValidationWorkbench.vue`（按 batchId）零改动。源硬编码处有**两处**（非一处）：暂存浏览页 `StagingBrowser.vue`（tabOptions/COLUMN_CONFIGS，仅 jt/st/yd）+ 首页 `CardFlowHome.vue`（stagingStatsMap `:144-148,324`）。
- **S2/S4 现状**：`ApplyCommonCostAllocation`（`AmoebaPLService.cs:1466-1503`；引擎 `Allocate(叶, 全额, scopeVol, fullVol)=全额×scopeVol/fullVol`）已就位，**内部对 scopeVol/fullVol 各调一次 `ComputeVolumeBasis`（`:1490-1492`）**；现 `ComputeVolumeBasis(points, matched, sendId, deliverId)`（`:1422-1431`）两侧 deliver 都从 `matched` 取（恒 0，待改）。S4 已加跨月周/全期告警兜底（`e3ea2bc`）。

## 2. 总体架构

```
派件 Excel(每网点/每日) ──上传──> [CardFlow 通用导入] ──> STG申通派件日明细
                                                              │
                                  AggregateDeliverVolume(起,止,组织, scope→F归属网点编号)
                                                              │  SUM(F基础派费收费件量)
                                                              ▼
  阿米巴日/周报 ── ApplyCommonCostAllocation ── ComputeVolumeBasis(scoped/full 各注入一个 deliver)
                       (S4 当月基线 / S2 同期基线)
```

`STG申通派件日明细` 是 Part A（导入）与 Part B（取数）的解耦接口。**取数过滤列用系统列 `F归属网点编号`（经 OutletResolver 规范化，与 billing `dp.SiteCode`/`scope.Outlets` 同源），不是 Excel 业务原始列 `F网点编号`。**

### A2. 迁移 + 导入源种子（`CardFlowSeeder` 新 `MigrateV23`）

> ⚠ 新步骤版本号 = **23**（当前最高 V22 + 1，`MigrationRunner.ValidateSteps` 强制连续）。既有迁移不可删。下文 `MigrateV23` 指同一新步骤。

1. **建表**：`IF NOT EXISTS(sys.tables…) CREATE TABLE [STG申通派件日明细](…全部系统列 + 业务列 + 索引…)`，**照搬 `MigrateV9` 的 `STG_出港运费` 写法**（含 `F原始行号`/`F其他列数据`/`F业务主键`/`F流水号 NULL`）。索引同样用 `IF NOT EXISTS(sys.indexes…)` 守护。列名/类型须与实体 Config **逐字一致**。
2. **导入规则** `CF自动插件_规则`（新 FID，`F类型编码=N'excelInput'`，`F组织ID`=**目标组织**）。
   - **⚠ F组织ID 不是写死 192**：现有 rule/流程定义 F组织ID 全硬编码 192（`:335/342/349/91-99`），匹配 `BuildRuleCandidatesAsync` 严格 `fd.FOrgId==orgId` 无 0 兜底（`BatchTriggerService.cs:427`）。**派件 rule/流程的 F组织ID 必须 = 实际上传派件文件的组织**（实施前确认汇华/太仓归属组织号）；多组织各种一套。错配则上传后静默不进流程、无报错。
   - `F规则配置JSON`：`targetTable="STG申通派件日明细"`、`outputMode="stg"`、**`headerRow=1`、`dataStartRow=4`**（示例前 3 行重复表头，**不可沿用 rule 3002 的 dataStartRow=2**，否则 row2/row3 表头被当数据导入污染表）；`totalRowDetection` 保留（"合计/总计"兜底）。
   - `columnMapping`：每项 `{excelColumn: Excel中文表头, dbColumn: 对应F列}`，覆盖全部 23 列。**Excel 表头须与真实文件逐字一致**（示例多行重复表头 + `&` 等特殊字符，按 row1 真实表头核对）。
   - `decimalFields`：所有件量/金额列。**不配 `dateFields`**：`ExcelInputPlugin` 不解析该键（惰性、无害但无用）；`F结算日期` 落 `date` 列靠 `SqlBulkCopy` 隐式转换（示例值 `20260511` yyyyMMdd），**实施时用示例文件实测该列能落 DATE**（既有 EF 实体型 STG 的 DateTime 列同路径已生产可用）。
   - `columnIdentifier`/`fullColumnIdentifier`：选**派件独有特征列**（含 `基础派费收费件量`、`业务员编码` 等），已核实与既有 3001/3002/3003 特征列互斥、不会误命中。提供 `fullColumnIdentifier` 走第一轮精确匹配降歧义。
   - **去重配置（方案 B）**：`crossBatchDedupEnabled=false`——**不配 `crossBatchDedupFields`**，最终去重完全交给 A1 的 4 字段唯一索引。
     - 去重语义：**首次导入正常；重复导入同 (组织,日,网点,业务员) → 唯一冲突 2601 → 插件 fallback（`:611-672`）的 NULL-JOIN 删不掉 → catch 放弃整批**。即"重复导入整批失败"，用户须**先撤销旧批次再重导**。派件场景"单文件=一网点一天全量、不增量补导"可接受此语义；需在导入说明/错误提示体现。
     - 同批次内重复仍由 `ComputeBusinessKey`（全 keyFields SHA256，`:978-989`）拦截。`keyFields=结算日期,网点编号,业务员编码`（Excel 中文名，供业务主键哈希；可 3 字段，不受两字段约束）。
     - 〔备选方案 A，本版不采用〕若后续需支持同网点同日增量补导：迁移加合成列 `F去重键`（`结算日期|网点编号|业务员编码` 单列），`crossBatchDedupEnabled=true`、`crossBatchDedupFields=["F去重键"]`（**DbColumn，带 F 前缀**），唯一索引改建在 `F去重键` 上——但须先验证插件能否经 `transformRules` 把合成列灌进 DataTable 参与去重（当前未验证）。
3. **流程（从零新建三件套）**：派件是全新流程，无可挂靠的既有定义/版本，必须新建：
   - `CF卡片流程`(CfFlowDefinition：新 FID，`F状态=published`，`F组织ID`=目标组织，`F匹配规则`={fileNamePattern 如 `*派件*`/`*进出港*` 作回退})。
   - `CF流程版本`(CfFlowVersion：新 FID，`F是否当前版本=1`，关联上面流程定义)。
   - `CF流程节点`(CfStageDefinition 首节点：`F排序号`=最小、`F类型=auto`、`F处理粒度=batch`、`F插件注册ID`=ExcelInput(=1)、`F插件规则ID`=新规则 FID)。
   - ⚠ `BuildRuleCandidatesAsync` 四表 JOIN 要求 `FStatus=='published' && FIsCurrentVersion && 首节点 F类型编码=='excelInput'`（`BatchTriggerService.cs:427-431`）缺一不自动匹配。FID 分配：流程/版本/节点/规则四套新 FID，实施时复核未被占用（种子普遍 `IF NOT EXISTS` 幂等）。
   - **纯导入，不接 Pricing/Cost/AutoVoucher 节点。**

## 4. Part B — 报表取数（Finance / `AmoebaPLService`）

### B1. `AggregateDeliverVolume(startDate, endDate, orgId, scope)`

原始 SQL（参数化，平行 `AggregateBillingRevenue` `:2086`）查 `STG申通派件日明细`：
```
SELECT ISNULL(SUM([F基础派费收费件量]),0)
FROM [STG申通派件日明细]
WHERE [F结算日期] >= @start AND [F结算日期] < @endExclusive   -- 半开，跨月周按周内实际天精确
  AND [FOrgId] = @org AND [FIsRevoked] = 0                    -- 按组织(不按账套，见 §5)
  [AND [F归属网点编号] IN (@outlets…)]                        -- scope.Outlets 非空时
```
- **⚠ 过滤列用系统列 `F归属网点编号`（规范化），不是 Excel 业务列 `F网点编号`**：`scope.Outlets`(= billing `dp.SiteCode` = 计费结果 `[F归属网点编号] AS SiteCode` `:2117`)是规范化网点编号；派件 Excel 原始 `F网点编号`(如 150049)若与之编码不一致会裁空/裁错。两侧必须同为 `F归属网点编号` 体系（由 OutletResolver 导入时写入系统列）。
- **按组织、不按账套**：与 `AggregateBillingRevenue`（仅 `[F组织ID]` 过滤 `:2126`）一致。
- 半开区间：`endExclusive = 末日.AddDays(1)`（与 billing/voucher 统一）。返回 long。
- ⚠ 原始 SQL `SqlQueryRaw` InMemory 不可测，靠 SQL Server 冒烟 + 范围计算抽纯函数边界覆盖（§B5）。

### B2. `ComputeVolumeBasis` + `ApplyCommonCostAllocation` 改造（一并改，勿只动签名）

deliver 改由调用方传入；**因 `ApplyCommonCostAllocation` 内对 scopeVol/fullVol 各调一次 `ComputeVolumeBasis`（`:1490-1492`），必须把两个 deliver 同时穿透**：
```
ComputeVolumeBasis(points, matched, sendId, deliverCount)        // deliver 直接传入
  send = billingSend>0 ? billingSend : matched[sendId]?          // send 仍 billing SUM(WaybillCount) 兜底
  deliver = deliverCount

ApplyCommonCostAllocation(…, scopedDeliver, fullDeliver)         // 新增两个 long 入参
  scopeVol = ComputeVolumeBasis(scopedPoints, scopedMatched, sendId, scopedDeliver)
  fullVol  = ComputeVolumeBasis(fullPoints,   fullMatched,   sendId, fullDeliver)
```
- ⚠ `fullPoints` 二次 `MatchDataPointsToPLItems`（`:1479-1483`）的 deliver 旧路径改注入后**对 deliver 失效**——只替换 deliver 注入，`fullMatched` 仍供 fullTotals/send 兜底，不可整段删。
- 既有 S2 单测随签名更新（且现有单测全是 `ComputeVolumeBasis` 孤立用例，不覆盖 full 分支 → 只改签名能全绿但 baseline deliver 仍 0，故必须连 `ApplyCommonCostAllocation` 一起改，见 §B5）。

### B3. 调用方接入（S2/S4 路径 `:331/338`）

调用方先算两侧 deliver，注入 `ApplyCommonCostAllocation`：
- **scoped deliver** = `AggregateDeliverVolume(本期起止, 组织, scope网点)`。
- **baseline/full deliver** = `AggregateDeliverVolume(基线起止, 组织, 全网点)`——S4 基线=所属当月、S2 基线=同期。
- caller 持 `periods[i]`/`granularity`/`request.Scope`，经 `PeriodToDateRange`/`PeriodContainingMonth` + `scope.Outlets` 推出两组日期+网点。

**根治三问题**：deliver 有源（非 0）；按网点裁（scope 感知，过滤 `F归属网点编号`）；半开范围求和（跨月周按周内实际天精确）。S4 的跨月周告警因此可移除或降级。

## 5. 数据流与口径

- **件量量纲**：send=billing 出港运单（`COUNT`/`WaybillCount`），deliver=派件结算 `SUM(基础派费收费件量)`。二者均为「件/单」级业务量、同组织维度明细，作为公共费分摊的相对权重（scope 占全口径份额）可比；total=send+deliver。⚠ 二者非同一物理动作（出港 vs 派件），total 基数语义是"该 scope 出港+派件总作业量占全机构份额"——分摊权重用途下成立，**§5 已显式记录此口径，不作绝对件数等价解释**。
- **按组织隔离（账套决策）**：派件量按 `FOrgId` 取数/去重，**不含 `F账套ID`**。理由：派件量是物理业务量、不随财务账套拆分；billing 侧 `AggregateBillingRevenue` 同样只按组织（计费结果表无账套维度），send/deliver 同口径自洽。`F账套ID` 仅入库（上传带则写、不带 NULL）。**前提**：同组织多账套时派件量合并为预期（同组织同日同网点同业务员物理上只一行，唯一索引保证谁先导谁占、重复导入整批失败）。〔若业务要求按账套隔离，则唯一索引/取数 WHERE 均须纳入 `F账套ID`——本版未采用。〕
- **方向（Direction）维度（澄清，非缺陷）**：deliver 天然进港。现有 total/deliver 基数公共费叶（FID 661/665-670 `FinanceSeeder.cs:2149,2153-2158`）挂在「公共分摊」Tab（FID 663，综合方向），方向 scope 过滤（`ApplyScopeFilter` `:267,1385,1357`）会整体剔除 → **deliver 接入不会让 OUT 子报表分母混入进港件量**。仅当未来把 total/deliver 基数叶直接挂到出港/进港 Tab 下，才需重评方向口径并补测试（记 §7）。
- **当月例外联动**：S4 当月基线经 `BuildPeriodDataPoints(当月,month)` 取公共费全额（含当月例外估值），deliver 基线则经 `AggregateDeliverVolume(当月范围)` 取**当月实际**——⚠ 隐含前提：deliver 基线**不含估值**、与含估值的公共费全额相除时，分子分母都用当月实际范围保证比例自洽；两口径独立但同期。

## 6. 范围边界 / 非目标

- 派费**金额**列只入库、**不接报表**（进港收入/成本仍走凭证）。
- **indicator 展示改源 + 历史手工数据处置 v1 不做**（原 §B4 延后，见 §B4）：v1 仅做分摊路径根治（§B2/B3）。`PerUnitValue` 单票均值的 deliver 分母同样依赖手工票量，一并随展示改源延后评估。
- 多维 scope 中派件量**主要按网点裁**（`F归属网点编号`）；方向/项目维度 v1 不特殊处理（方向口径见 §5 澄清）。
- 去重用**方案 B**（4 字段唯一索引 + 整批失败语义）；合成列方案 A 为备选、本版不实现。
- 不为派件接 Pricing/Cost/AutoVoucher 等后续节点（纯导入落库）。
- 账套维度 v1 按组织合并（见 §5）。

## 7. 风险与注意

1. **STG 前缀建表陷阱**：必须显式迁移 CREATE TABLE（`ExcludedPrefixes` 含 STG），否则存量库无表、导入即失败。
2. **建表 DDL 三列 + 来源**：DDL 以 `MigrateV9`/`STG_出港运费` 为模板（含 `F原始行号`/`F其他列数据`/`F业务主键`），勿照 `StgShentongOutbound` 实体（缺三列→导入即炸）。`F流水号` NULL。
3. **列名逐字一致**：实体 Config / 迁移 DDL / 规则 `columnMapping.dbColumn` 三处列名完全一致；Excel 表头与 `columnMapping.excelColumn` 逐字一致（多行重复表头 + `&`，按 row1 核对）。
4. **去重方案 B 语义**：重复导入整批失败、需先撤销旧批次再导；导入提示需体现。不配 `crossBatchDedupFields`（配了也只比前两列、且 Excel 名会静默禁用）。
5. **dataStartRow=4 / headerRow=1**：示例前 3 行重复表头；不可沿用 rule 3002 的 dataStartRow=2。
6. **F组织ID**：rule/流程定义的 F组织ID = 实际上传组织（非写死 192），否则上传静默不匹配；多组织各种一套。
7. **取数过滤列 `F归属网点编号`**：与 billing/scope.Outlets 同源，勿用 Excel 原始 `F网点编号`。
8. **deliver 改造面**：必须同改 `ApplyCommonCostAllocation`（scopedDeliver/fullDeliver 双注入）+ 调用方两次 `AggregateDeliverVolume`，只改 `ComputeVolumeBasis` 会漏 full 侧 → baseline deliver=0、比例退化。
9. **撤销走 F批次ID 物理删除**：`RevokeByDataScopeAsync` 对派件恒 no-op；软撤销不级联 STG 行。
10. **方向维度**：现架构下安全（公共分摊叶在综合方向被整体过滤）；未来若把 total/deliver 基数叶挂到方向 Tab 下需重评。
11. **AggregateDeliverVolume 原始 SQL**：InMemory 不可测，靠 SQL Server 冒烟 + 纯函数边界。
