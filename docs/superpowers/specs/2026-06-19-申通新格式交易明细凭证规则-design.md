## 1. 新旧格式差异（实测）

| 语义 | 旧格式（17 列，sheet `sheet1`） | 新格式（17 列，sheet `data`） |
|---|---|---|
| 凭证日期 | 业务日期 / 记账日期 | **会计日期** / 扣付日期 / 结算日期 |
| 网点 | 网点编号 / 网点名称 | 网点编码 / 网点名称 |
| 交易类型 | 业务类型 | 交易类型 |
| 费用编码 | `费用科目编码`（`1122.11`/`2202.13` 带点 2 级） | `费用编码`（**6 位纯数字** `030201`/`060205`；资金类用 `03.xx` 点码 + 空码） |
| 收付类型 | `费用科目收付类型` | `费用收付类型` |
| 进出港 | 无（靠业务逐项推断） | **`进/出港标识` 显式列**（综合/出港/进港/空） |
| 其他 | 报表业务日期 / 联系方式 / 账单类型 | 联系人 / 银行账号（无账单类型/报表业务日期） |

**编码前缀（2202=应付 / 1122=应收）判收付方向的旧逻辑失效**——新编码 6 位不编码方向，方向改由两个显式列 `费用收付类型` + `进/出港标识` 承载。**收付类型铁律仍成立**。

### 1.1 文件格式坑（已实测验证，风险已解除，见 §5 R1）
4 个 `.xls` 文件**实为 xlsx**（PK/OOXML zip 头，扩展名骗人）。其中**南郊/浏河是 `inlineStr` 内联字符串 + `dimension="A1"`（未声明真实范围）的流式导出**（申通系统原始导出常态），城区/沙溪是被 Excel 另存过的标准 xlsx（sharedStrings + 正常 dimension）。`dimension="A1"` 会让"信 dimension"的解析器（如 openpyxl read_only）把数据**静默读成 0 行**。

## 3. 架构设计（A–G，全部 additive，旧链零改动）

### A. 路由 — 列头自动分流（已验证安全）
`BatchTriggerService.MatchFlowDefinitionsAsync` 三轮短路：**①fullColumnIdentifier 精确（整列集排序相等）命中即返回 → ②columnIdentifier 包含 → ③fileNamePattern 回退**。`BuildRuleCandidatesAsync` 仅取**当前版本**（`FIsCurrentVersion`）的首节点 `excelInput` 规则。
- 新导入规则 `fullColumnIdentifier` = 新 17 列；旧规则 20 `fullColumnIdentifier` = 旧 17 列。两者列集不同 → **第一轮即各自唯一命中、互不串**，短路在文件名匹配之前。
- 虽然新旧文件名都含"交易明细"（旧流程 2257 `F匹配规则={"fileNamePattern":"*交易明细*"}`），但**第三轮文件名回退永不触发**（第一轮已命中）。
- 双保险：新导入规则 `columnIdentifier` = 新格式独有列 `会计日期,费用编码,费用收付类型,进/出港标识`（旧格式均无），即使第一轮失配，第二轮也唯一命中新流程。

### B. STG 表变更 — 复用 `STG申通总部交易明细` + 加 2 可空列
现有表/实体 `StgShentongHqTx` 无收付类型/进出港列。新增（均可空，旧格式导入留空、零影响）：
- `F费用收付类型` `nvarchar(50)`
- `F进出港标识` `nvarchar(20)`

**STG 前缀表不被 SchemaAutoSync/CreateMissingTables 自动建/改**（各自 ExcludedPrefixes 含 "STG"）→ 必须在 `CardFlowSeeder` 写**显式 `ALTER TABLE … ADD`**（`IF COL_LENGTH(...) IS NULL` 守卫幂等）。同步在 `StgShentongHqTx.cs` 实体 + 其 `IEntityTypeConfiguration` 加这两个属性/列映射（供全新库 `CreateTablesAsync` 建出）。

新日期列：`会计日期→F业务日期`（凭证 dateField）、`扣付日期→F记账日期`、`结算日期/联系人/银行账号`未映射 → 自动入 `F其他列数据` JSON（凭证不需要，YAGNI）。

### E. 自动凭证规则（新 `AutoVoucher`，FID=3131，蓝本=规则 21，结构逐字对齐）
顶层：
```
mode=rulesBased ; version=2 ; voucherWord=记 ; dateField=F业务日期 ;
stagingTable=STG申通总部交易明细 ; accountSetId=2 ;
keyFields=[F科目编码,F费用名称,F业务摘要] ; unmatchedAction=createDraft ;
draftPlaceholderAccountId=700044 (1901 待处理财产损溢) ;
matchingLayers={ exactMatchField:F科目编码, categoryField:F费用名称, summaryField:F业务摘要 }
```
**ruleGroups**：每个损益编码一组（≈150 组），`exactCodes=[6位费用编码]`，`amountAggregation=ROW`，2 分录行——**金额列由收付类型决定**（与规则 21 完全一致）：

| 收付类型 | Line1（损益） | Line2（对手） | amountField（两行同列） | business_direction |
|---|---|---|---|---|
| 总部应收（成本/费用） | **借** 品牌版成本科目(fid) | **贷** 220201 总部应付(700125) | 见下 amountField 口径 | 每组固定（见下） |
| 总部应付（网点收入） | **贷** 品牌版收入科目(fid) | **借** 220201 总部应付(700125) | 见下 amountField 口径 | 每组固定 |

**⚠️ amountField 口径（与规则21不同，关键修正）**：规则21 按收付类型固定取单列（应付→收入列/应收→支出列）。新格式有 **12 组金额主额落在相反列**，沿用单列会**漏记约 386 万**（小件员直达派费 168万/政策返利 143万/对客理赔 35万/政策互惠 22.8万 等）。本设计每组 amountField **取主额所在列**（配置期按真实数据定，见映射草案 §0），方向仍按收付类型。其中部分还涉及「收付类型↔经济实质」矛盾（如 060408 政策返利 标"应收/成本"但实为网点返利收入），需财务单独裁决科目侧+方向——列入映射草案 §0 待复核。

辅助核算（与规则 21 逐字一致）：
- 损益行 3 维：`outlet`(dynamic, sourceField=F网点编号, matchBy=code) + `express_brand`(fixed fid16/ST) + `business_direction`(fixed OUT/IN/CMB)。
- 220201 行 1 维：`express_brand`(fixed fid16/ST)。
- `business_direction` 固定值在配置期从该编码真实「进/出港标识」推导（综合→CMB / 出港→OUT / 进港→IN）；唯一一码两向 011116 取主方向（出港，43 行多为出港）。
- `business_unit`（经营单元）本轮不配（同规则 21，待"网点→经营单元"映射）。

**资金往来类（~14 组合/1592 行：调账/充值/取款，含混入的 03.73 网点三件理赔 等）不建组** → 由 `unmatchedAction=createDraft` 落待补录草稿（占位科目 1901），人工补账。与规则 21 处置一致；资金另一端银行流水线本就空壳。

### F. 映射推导方法（产出独立草案供逐条复核）
按 **收付类型铁律 + 13 性质组 + 一码一名**，把 165 编码（剔除资金类）→ 现有品牌版科目（fid 见 §账套2 科目快照）。桥接键 = **费用名称**（新数据费用名 ↔ 规则 21 已确认科目）。
- 名称能对到规则 21 映射的 → 直接复用其品牌版科目 + BD 口径。
- 规则 21 没有的新名称（如 业务治理费/治理扣费/集运直邮补贴/申咚咚激励金/小件员直达派费/政策互惠 等）→ 按性质归到对应品牌版叶（出港收入/派费收入/考核激励/操作费/政策返利/期间费用 等），标 `low` 置信 + "待财务确认"。
- 成本三分：主营结算 5401 / 期间费用 56xx（租金 560204/水电 560205/利息 560301/通讯 56020901/手续费 560302）/ 税金 5403。

产出 `2026-06-19-申通新格式映射草案.md`(+`.xlsx`)：每行 `费用编码 | 费用名称 | 收付类型 | 进出港 | 行数/收入/支出 | → 品牌版科目码+名(fid) | BD | 置信 | 备注`。

## 5. 风险与验证

- **R1 解析器读流式 inlineStr/dimension="A1" — 已实测解除。** 临时探针测试调生产 `ExcelParserService.ParseAsync` 读全 4 文件，全部通过（含南郊/浏河）：行数达标、`费用编码/费用收付类型/进/出港标识` 列齐、费用编码非空行充足。结论：`SniffExcelFormatAsync` 魔数嗅探（PK→.xlsx 走 MiniExcel）+ MiniExcel 逐行读（不信 dimension）能正确处理申通原始导出，**无需任何解析器加固**。（探针验证后已删除，实现期以端到端导入行数核对为永久门禁。）

- **R3 金额方向与收付类型矛盾（重要，已升级处置）**：12 组金额主额落在与收付类型相反列，单列口径漏记约 **386 万**。处置=amountField 取主额列（§E）；其中涉经济实质矛盾者（政策返利/政策互惠/对客理赔/小件员直达派费 等）列入映射草案 §0 待财务裁决科目侧+方向。净额双显著（一口价补贴/业务治理费）主额仍在取数列、残差反向小额沿用单列。

## 7. 非显然事实（实现必读）

- 路由只认**当前版本**首节点 `excelInput` 规则的列标识；`F类型编码` 比较在 SQL CI 排序下 `ExcelInput==excelInput`（旧规则 20 是 `ExcelInput`，新规则用 `excelInput` 皆可）。
- `decimalFields` 用 **dbColumn** 名（规则 20 实证：`F发生额收入` 等）；`keyFields`/`columnMapping.excelColumn`/`fullColumnIdentifier` 用 **excelColumn**（表头原文，含 `进/出港标识` 的斜杠、`发生额(收入)` 的括号）。
- 自动凭证只认 `accountId`(fid)；`accountCode` 仅跨账套复制重映射用，引擎出凭证不读。品牌版 fid 段：收入 50010101~ = 700304~、成本 54010101~ = 700341~、对手 220201=700125、占位 1901=700044。

- 迁移版本严格连续 +1，不可改已执行版本；STG 前缀表存量库不自动建/改列，必须显式 ALTER。
- 4 网点 outlet 辅助项已存在（320288 城区 fid7/320319 沙溪 fid8/321426 浏河 fid9/321992 南郊 fid10，账套2/org192）；express_brand ST=16、business_direction OUT=13/IN=12/CMB=14（全局账套0）。
