## 3. 详细设计

### F1 启停状态正确性（核心 bug，P1）

**问题**：前端开关绑 `record.isEnabled`，但后端返回字段是 `enableStatus`（`AccountService.cs:560`/`:585` 的 `EnableStatus`，经全局 camelCase 序列化）。字段不匹配 → 所有科目开关 `:checked` 恒 `undefined` → **全部显示"关"**。更糟：`handleStatusChange` 因显示态恒为关，用户点击时 `val=true`（视为"启用"）→ `if(!val)` 不成立 → **跳过停用二次确认** → `toggleAccountStatus(id)` 后端是**纯翻转** → 把启用中的科目**静默停用**，使 6-19 完成的"停用三拒绝确认"形同虚设。

**证据**：模板 `web/src/views/finance/AccountManage.vue:108`、列定义 `:355`、`AccountNode` interface `:392`；`loadData` `:583` 原样赋值、`flattenAccounts` `:424` 用 `{children, ...rest}` 透传不重命名；`handleStatusChange` `:779`、`doToggleStatus` `:793`（失败仅 `console.error` 不刷新）。

**方案**：
1. 三处 `isEnabled` → `enableStatus`：模板 `:108` 的 `:checked`、列定义 `:355` 的 `dataIndex`/`key`、`AccountNode` interface `:392`。
2. `doToggleStatus` 改为 `finally { loadData() }`：成功失败都刷新，保证开关视觉与后端真实状态一致（后端拒绝停用时开关回弹原态）。
3. 后端 `ToggleStatusAsync` 不改（纯翻转语义本身正确）；前端 `val` 仅用于"是否弹确认 + 提示文案"，字段修正后确认分支自然恢复正确。

**验收**：开关如实反映 `enableStatus`；停用"启用中"科目时弹确认；后端拒绝停用后开关回弹为"开"。

### F2 读接口账套权限（防越权，P1）

**问题**：`AccountController` 5 个读接口仅类级 `[Authorize]`（`AccountController.cs:22-94`），`accountSetId` 来自 query，service `GetTreeAsync` 只按传入账套过滤（`AccountService.cs:46-47`）无授权校验 → 任意登录用户传别的 `accountSetId` 即可越权读该账套科目树/期初余额/选择器。`SubjectView`(`accountset:subject:view`) 常量已定义（`AccountSetPermissions.cs:28`）却**零使用**。

**关键约束**：账套角色权限（`FinAccountSetRolePermission`）是**运行时在账套授权页配置的数据，无代码种子**（`AccountSetAuthorizationService.HasPermissionAsync` 走"用户→账套授权→账套角色→权限码"链，`:57-68`）。`SubjectView` 从未被任何特性要求过，配置者很可能只勾了 `SubjectEdit`。若直接加 `[Require(SubjectView)]`，这些**非管理员会连科目页都打不开**。

**方案**：
1. **过滤器扩展为 OR**：`RequireAccountSetPermissionAttribute` 构造由 `(string permissionCode)` 改为 `(params string[] permissionCodes)`，校验时**任一**权限命中即放行（`RequireAccountSetPermissionAttribute.cs:57-79`）。`params` 兼容所有现有单参数调用点。
2. **5 个读接口加特性**：`GetTree`/`GetSelectorList`/`GetByAuxType`/`GetInitialBalances`/`GetById` 加 `[RequireAccountSetPermission(AccountSetPermissions.SubjectView, AccountSetPermissions.SubjectEdit)]`（"查看权 或 编辑权"，编辑权蕴含查看权）→ 零数据迁移、零误伤、对未来新角色自然成立。管理员由过滤器 `IsAdmin` 放行（`:35-39`），不受影响。前端已统一注入 `X-AccountSet-Id` 头（`web/src/api/request.ts:67-69`），过滤器从头或 query 参数都能取到账套。
3. **GetById 彻底校验账套匹配**（用户决策）：科目 id 全局唯一、`LoadAccountAsync` 不按账套过滤（`AccountService.cs:102-104`），仅加特性仍可"传 A 账套头 + B 科目 id"读到单条详情。故：
   - controller `GetById(long id, [FromHeader(Name="X-AccountSet-Id")] long accountSetId = 0)`；
   - service `GetByIdAsync(long id, long accountSetId)` 内校验 `account.FAccountSetId == accountSetId`，不符则按"科目不存在"返回 `null`（不泄露存在性）；
   - 账套通过**显式参数**传入，service **不读** HttpContext（与 F18 协同，避免重新引入 `_httpContextAccessor`）。

**验收**：非授权账套调任一读接口返回 403；`GetById` 跨账套返回"科目不存在"；持 `SubjectEdit`（无显式 `SubjectView`）的非管理员仍能正常读；管理员不受影响；现有写接口调用点编译通过。

### F6 创建/更新输入校验（P2）

**问题**：`CreateAsync`（`AccountService.cs:113-169`）校验了编码（数字/层级/前缀/长度/唯一）与类别白名单，但 `request.Name` 无非空校验（`:174` 原样写入）、`request.BalanceDirection` 无取值校验（`:176` 原样写入），可入"空名科目""非法方向"脏数据；方向是后续报表借贷归集依据。

**方案**：
- `CreateAsync` 增加：`string.IsNullOrWhiteSpace(request.Name)` → 拒绝；`request.BalanceDirection` 必须 ∈ `{"借","贷"}`。
- `UpdateAsync` 的 `Name` 同样加非空校验（一致性）。
- **不做**"类别↔方向一致性"硬校验——备抵科目（累计折旧/摊销/减值，方向=贷）等是合法例外，硬卡会误伤（见 memory 关键事实校正）。

**验收**：空名 / 非法方向的创建、空名的更新被拒绝并返回友好消息。

### F16 `GetByAuxType` 精确匹配（P2）

**问题**：`a.FAuxiliary.Contains(auxType)`（`AccountService.cs:487`）对逗号拼接的辅助编码串（`UpdateAccountAuxiliaryAsync` `:519` 用 `string.Join(",", ...)` 写入）做子串匹配，存在前缀误命中（如 `Contains("dept")` 命中 `department`）。

**方案**：改为 `("," + a.FAuxiliary + ",").Contains("," + auxType + ",")`（首尾补逗号后整项匹配，EF 可翻译为 LIKE）。

**验收**：`auxType` 为某编码时不再误命中以其为前缀的其它编码。

### F18 死代码清理（P2）

**问题**：`GetCurrentOrgId()`（`AccountService.cs:33-38`）与其依赖的 `_httpContextAccessor` 字段（`:17`，构造注入 `:24/:30`）仅互相引用，全模块无其它调用；科目账套级共享、本不按组织过滤，该方法的存在还会误导后来者。

**方案**：删除 `GetCurrentOrgId()` 方法 + `_httpContextAccessor` 字段 + 构造函数参数 `IHttpContextAccessor` + 不再需要的 `using Microsoft.AspNetCore.Http`。（F2 的 GetById 账套校验走显式参数，不需要 accessor。）

**验收**：build 绿；全模块无 `GetCurrentOrgId`/`_httpContextAccessor` 引用残留；DI 正常解析（构造参数减少无需改注册）。

### F11 固定列生效（P2，前端）

**问题**：`<a-table>`（`AccountManage.vue:45-55`）未配 `:scroll`，而 `accountColumns` 的操作列 `fixed:'left'`(`:348`)、启停列 `fixed:'right'`(`:355`) 依赖 `scroll.x` 才生效，当前为空操作。

**方案**：给科目树 `<a-table>` 加 `:scroll="{ x: 'max-content' }"`。

**验收**：列宽溢出时操作列/启停开关分别吸附左右两端。

### F13 期初 Tab 按钮语义（P2，前端）

**问题**：PageHeader `#right` 的"新增"`:8`、"删除"`:11`、"更多"下拉 `:14-24` 始终显示（仅受 `SubjectEdit` 权限），在 `currentCategory==='initial'`（期初值录入视图）下语义错位，"新增"还会回退打开默认"资产"建科目弹窗（`handleAdd` `:606-614`）。

**方案**：三者增加 `currentCategory !== 'initial'` 条件——"新增"/"删除"并入既有 `SubjectEdit` 的 `v-if`，"更多" `a-dropdown` 加 `v-if`。

**验收**：切到期初值视图时不再显示新增/删除/更多。

### F14 齿轮图标（P2，前端）

**问题**：单位列在有值时渲染 `{{ record.unit }}<SettingOutlined .../>`（`:101-104`），齿轮无 `@click`/无 tooltip，强烈暗示"可点击配置"却无反应。

**方案**：**去掉该图标**（用户决策）；数量核算的编辑入口在操作列的编辑弹窗。同时清理对应的 `.unit-icon` 样式与（若因此不再使用的）`SettingOutlined` import。

**验收**：单位列只显示单位文本或 `-`，无误导图标。
