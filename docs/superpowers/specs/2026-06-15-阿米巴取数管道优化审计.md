# 阿米巴取数管道与字段格式优化 — 审计报告

> 生成方式：多 agent 工作流审计（68 个 agent，54 条发现，对抗核验后确认 37 条）。
> 范围：`AmoebaPLService` 取数管道 + `FinAmoeba*` 字段/数据模型。
> 日期：2026-06-15。三条最高风险项（P0-1 出港收入恒0、P1-4 orgId 口径分裂、P1-6 月末边界）已人工复核证据。

## 1. 总览

**一句话判断**：阿米巴损益取数管道“能跑、但口径脆弱”，字段模型“半落地正交列遍地、新旧双轨未收口”。当前唯一活跃入口（多期主报表）依赖一套“科目前缀 StartsWith + 树前序先到先得独占消费”的隐式归属约定，**业务方向被藏在 Tab 结构里靠五层启发式反推**，而真正该承载方向的字段（`F指标方向范围`、V2 filters 的 `business_direction`）要么是死列、要么种子里恒空。这套机制在现网“全限定末级码 + Tab 隔离”的旧模板下尚能侥幸跑对，但**目标 72 项 CSV 改用“粗码 + 业务方向/部门”消歧后，同码碰撞会从隐患变成必然**。

健康度评级：
- **取数正确性**：🔴 高风险（同码独占吞并、出港收入恒 0、收入符号翻负、月末漏单、红冲/未过账未排除）
- **字段模型**：🟡 中风险（4+ 个正交列半落地、新旧双轨、靠一次性迁移启发式反推）
- **健壮性/越权**：🟠 中高（递归无防环、orgId 可越权且四源口径分裂）
- **性能**：🟢 低风险（均为常数级浪费，无量级问题）
- **前端契约**：🟡 中（多期对比在最关键的 Tab/全局利润行上反而失效）

**P1-2 源数据未打 business_direction 标——一等公民真正缺口**

**P1-3 工资靠 department 而非 business_direction 消歧**

**P1-4 orgId 可越权 + voucher 不按 orgId 过滤→跨组织错位拼接**

## 3. 与种子方案 A/B/C 的关系

> 审计反推的三方案命名（与早先沟通口径不同，下文以此为准）：
> - **方案 A — 全限定末级码物理拆码**：每损益项配唯一末级科目，靠科目码+Tab 区分方向，不依赖业务方向辅助核算（≈ 现网旧模板做法 + 真正共享项归公共/综合）。
> - **方案 B — 粗码 + 业务方向辅助核算**：共享粗码，靠 V2 filters 的 business_direction（成本）/department（工资）消歧，即“业务方向一等公民”。
> - **方案 C — 混合**：能用末级码区分的走 A，科目层确实共享同码方向二分的走 B。

| 优化 | 对方案选择的影响 |
|---|---|
| **P1-1 业务方向一等公民** | 决定性。证实 B 的匹配侧零成本（代码已支持、科目已声明、编码已注册）。但 B 可行性取决于 P1-2 源打标。 |
| **P1-2 源数据打标** | B 的硬闸门。源分录不带 business_direction 则 filter 配了也匹配空→共享码叶恒 0。B 当前不可直接落地的唯一根因。 |
| **P1-3 工资 department 双维** | B 须同时支持 business_direction(成本)+department(工资)，且需主数据细分部门码。 |
| **P0-2/P0-3 同码独占+父吞子** | A 天然规避（末级码唯一不碰撞）；B 必须靠 P1-1+P1-2 才规避。A 短期更安全的核心原因。 |

**结论**：实施 P1-1+P1-2 后 B 完全可行且优于“给科目表加出/进港双末级码”（不撑大科目表，与设计文档一致）。前置链：`科目支持方向(已就位)→源自动打标(P1-2,缺失,L级)→种子写 filters(P1-1,S级)→匹配分流(已就位)`，唯 P1-2 是工程闸门。

## 4. 立即决策建议

> **本步（72 项损益项种子）选方案 A：全限定末级码物理拆码，正交列在 INSERT 显式写全，不依赖业务方向辅助核算；真正共享的成本（运输/物料/装卸/考核/客服赔付/工资）归“公共分摊(综合)按件量分到出进港”。**
> 因为 B 的唯一技术闸门——源凭证/账单 business_direction 自动打标（P1-2）——当前完全缺失且 L 级跨模块，且 P0-1/P1-5/P1-6 等取数地基未修。先用 A 立即正确出数，第 2 批完成源打标后把 13 性质组中科目层共享方向的项**增量切到 B（终态 C）**，工资一律走 department。
