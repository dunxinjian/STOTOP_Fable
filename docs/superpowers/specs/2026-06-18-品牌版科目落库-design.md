# 设计：品牌版会计科目落库

- 日期：2026-06-18
- 范围：`STOTOP.WebAPI`（FinanceSeeder 种子/迁移）、`docs/superpowers/specs/seeder科目替换脚本.py`（落库代码生成工具）
- 类型：数据/种子重构 + 版本化迁移
- 状态：brainstorm 已通过，待写实现计划
- 前置依赖：6-16「FIN科目去组织作用域」（已落库，`FIN科目` 已无 `F组织ID` 列）
- 后续单列 spec：② 凭证规则按品牌版重配（规则21 + 其他三家）；③ 阿米巴损益项关联科目配置

---

## 1. 背景与问题

系统目标是「四家快递品牌（申通/韵达/极兔/中通/圆通）通用 + 阿米巴五周期经营报表」。其会计地基——科目体系——现状存在**三套并存且代码与库脱节**：

| 来源 | 科目形态 | FID 段 | 状态 |
|---|---|---|---|
| `FinanceSeeder.cs` MigrateV1（909 条，全新建库基线） | **网点版**（101201=申通城区；账套1石家庄/账套2太仓各网点） | 账套1=600xxx / 账套2=700xxx | 已在 master |
| `2026-06-15-新科目表.csv`（427 项）+ `科目覆盖迁移.sql` | **品牌版**（101201=申通预付款；网点降为辅助核算） | — | **只在 docs，从未落库** |
| 现有库账套2（430 项，replace_accounts.py 5-11 导入） | **网点版** | 200900~201329 | 当前测试库，凭证规则21 配套它 |

**网点版的致命缺陷**（2026-06-18 实测+对比确证）：
- 1012 预付款只能按申通网点组织，**装不下韵达/极兔/中通**；
- 收入/成本科目层把细分（考核类型/服务类型）铺成独立末级，**没有任何辅助核算维度**，分类还混乱（`540101 面单` 下塞了派费/政策返利/一口价）；
- 阿米巴要的「经营单元 × 业务方向(出/进港) × 品牌」维度科目层根本没有 → **阿米巴经营分析算不出来**。

**品牌版**把科目按性质聚合、细分维度全部移到辅助核算（`[辅:网点,业务方向,快递品牌,经营单元,客户,员工,项目]`），按品牌设预付款——正是四家通用 + 阿米巴所需。两账套当前均 **0 凭证**，是切换成本最低的时机。

## 2. 目标与非目标

### 目标
- 品牌版科目落到两个账套（账套1 石家庄、账套2 太仓），成为**唯一真相源**：全新建库与现有库都收敛到品牌版。
- **完整保留各账套现有的 1002（银行存款）/ 1012（其他货币资金/预付款）及其全部子科目**——这两段是各地真实开立的银行户与预存款户，不能用模板覆盖。
- `dotnet build` 通过；全新建库成功；现有库经版本化迁移收敛；不破坏 0 凭证现状。

### 非目标（明确排除）
- **不重配凭证规则**（规则21 的 88 个 accountId + 60 个 accountCode 会因 FID/编码变化失效）——属 ②spec。
- **不配置阿米巴损益项关联科目**（79 项 `F关联科目JSON`）——属 ③spec。
- 不改 `FIN账套`、`FIN科目模板`（表头）种子；不动 `FinVoucher/FinVoucherEntry` 等其它实体。
- 不在 1012 预设韵达/极兔/中通预付款户——预付款是实账户，由各账套按真实开户时再加。

## 3. 已确认决策

| 决策 | 选择 |
|---|---|
| D1 版本 | 品牌版（四家通用、维度走辅助核算） |
| D2 账套范围 | 账套1 + 账套2 都做 |
| D3 资金科目 | 完整保留各账套现有 1002/1012 及子科目（账套1=41银行+32预付款；账套2=13银行含韵达极兔+5申通预付款） |
| D4 落库方式 | A：改 FinanceSeeder 基线 + 加版本化迁移 V9 |
| D5 1002/1012 真相源 | **现有库**（用户维护的真实账户），非旧基线种子 |
| D6 FID 策略 | 删全部业务+资金科目重建，FID 统一重排（账套1=600xxx/账套2=700xxx）；1002/1012 保留**内容**而非 FID（0 凭证，FID 变无副作用，避免基线与现库 FID 分裂） |

## 4. 设计

### 4.1 组件1 · 落库代码生成（改造 `seeder科目替换脚本.py`）

现成脚本已实现核心逻辑（读品牌版 csv、跳过 1002*/1012*、提取保留段、合并重排 FID、生成 INSERT）。需三处改造：

1. **1002/1012 来源改为现有库**（D5）：从目标库 `SELECT … FROM [FIN科目] WHERE F账套ID IN (1,2) AND (F编码 LIKE '1002%' OR LIKE '1012%')` 导出真实账户，替代当前"从 FinanceSeeder.cs 提取"。
2. **去 `F组织ID` 列**（对齐 6-16）：生成的 INSERT 列清单与 VALUES 均不含 `F组织ID`（现 `FIN科目` 表已无此列）。
3. 输出两份产物：① 替换进 MigrateV1 的基线 INSERT 块；② MigrateV9 用的 INSERT 块（同一批语句）。

合并与排序：`品牌版业务科目(csv 除 1002/1012) + 各账套真实 1002/1012` → 按 `(len(编码), 编码)` 排序 → 账套1 FID 从 600000、账套2 从 700000 递增 → `F父ID` 按父编码回填。

### 4.2 组件2 · 基线种子改造（`FinanceSeeder.cs` MigrateV1）

- 用组件1产物替换 MigrateV1 中两账套的 `INSERT INTO [FIN科目]` 块（网点版 → 品牌版业务 + 真实 1002/1012）。
- `FIN账套` / `FIN科目模板`（表头）种子保持不动。
- 全新建库 `CreateTablesAsync` + MigrateV1 → 品牌版业务科目 + 各账套真实资金科目。

### 4.3 组件3 · 版本化迁移 `MigrateV9`（覆盖现有库）

`FinanceSeeder.Migrate` 的 steps 末尾追加（当前最高 V8）：

```csharp
new(9, "品牌版科目覆盖：删两账套旧科目重建(保留各账套真实1002/1012内容) (2026-06-18)", MigrateV9),
```

```
MigrateV9(ctx):
  if !IsSqlServer(ctx) return;
  事务内：
    DELETE FROM [FIN科目] WHERE [F账套ID] IN (1,2);
    INSERT <组件1产物：品牌版业务 + 各账套真实1002/1012，FID 600/700xxx>;
```

- 现有库账套2 业务科目 FID 是 200xxx（replace 版），DELETE 按 `F账套ID` 删，不依赖 FID。
- 幂等：MigrationRunner 按版本号只执行一次，记 `SYS迁移历史` Finance V9；版本号严格递增。
- 安全：两账套 0 凭证（已实测确认 `FIN凭证`/`FIN凭证分录` 账套1/2 均 0 行），无外键引用，删全部科目重建无副作用。

### 4.4 组件4 · 科目模板明细（条件式处理）

`FIN科目模板`（3 个：石家庄申通/太仓美申/快递行业标准）+ `FIN科目模板_明细`。实现第一步**先核实模板是否参与"新建账套"流程**：
- **若参与**（新建账套从模板实例化科目）：同步生成品牌版模板明细，保持模板与账套科目一致（参考 replace_accounts.py 旧逻辑：模板明细与科目对应）。
- **若不参与**（模板为历史残留、现流程不消费）：本 spec 不动模板明细，在 spec 末"已知偏差"记录，留待单独清理。

## 5. 连带影响与边界

- ⚠️ **凭证空窗**：本 spec 落库后，规则21 的 88 个 accountId（指向旧 FID 200xxx）与 60 个 accountCode（已重组消失）全部失效 → 申通自动凭证生成会失败/错配。故 **①(本spec) 与 ②(规则重配) 必须连续实施，中间不在生产使用导入**。本 spec 验证只在隔离/测试库进行。
- 阿米巴损益项（③spec）依赖科目落库完成后才能配关联科目；本 spec 不涉及。
- 预付款（往来户）映射因 1012 内容保留，②spec 重配时只需更新其 FID，科目语义不变。

## 6. 验证

- [ ] `dotnet build` 通过。
- [ ] **种子完整性**：MigrateV1 两账套 `INSERT INTO [FIN科目]` 每条「列数=值数」（脚本校验），无 `F组织ID` 列。
- [ ] **全新建库实测**（隔离库）：科目树正确——`1012` 各账套真实账户（账套2=申通城区…保留）、`5001/5401` 品牌版聚合且带辅助核算（`[辅:网点,业务方向,快递品牌,经营单元…]`）；启动无 SchemaAutoSync「多余列」告警。
- [ ] **现有库 V9 迁移**：科目收敛品牌版、1002/1012=各账套真实账户不丢、`FIN凭证` 仍 0 行、`SYS迁移历史` 记 Finance V9。
- [ ] **抽样核对**：账套2 `101201=申通城区`(保留)、`540102=派费`(11 末级)、`500101=出港收入`(挂全维度辅助核算)；账套1 `1012` 32 个申通网点预付款保留、`1002` 41 银行户保留。
- [ ] 账套数、各账套科目数与预期一致（品牌版业务科目数 + 各账套 1002/1012 子科目数）。

## 7. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 种子去列/重排出错（列值错位、父子 FID 错），build 查不出 SQL 字符串错 | **高** | 脚本生成 + 每行「列数=值数」校验 + 全新建库实测（§6） |
| 1002/1012 从现有库导出时账套1/2 串账或漏子科目 | 中 | 导出按 `F账套ID` 分别处理 + 抽样核对各账套子科目数（§6） |
| 改基线 MigrateV1（违反"只追加"惯例） | 中 | 末态收敛（新库走改后 V1、旧库走 V9）；沿用 6-16 同款权衡 |
| 模板明细与账套科目不一致 | 中 | 组件4 条件式处理：先核实模板是否参与流程再决定同步/记偏差 |
| ①②之间凭证空窗 | 中 | 限定测试库验证；①② 连续实施，不在该窗口跑生产导入（§5） |
