## 1. 子项分解（按风险与是否触碰 FlowEngineService 分组）

### 🟢 3a · 死成员纯删（零风险，类阶段1）
- `BatchTriggerService.MatchFlowDefinitionAsync`（:616，标 `[Obsolete]`，一行转发 `MatchFlowDefinitionsAsync`，**全库零调用方**）
- `QualityAnalysisPlugin.ValidateRowBuiltin`（:365，`private static`，**零调用方**）
- `ITransformEngine.Preview`（接口 `Services/Import/TransformEngine/ITransformEngine.cs:11` + `JintTransformEngine` 实现，**无调用方**）
- **文件**：上述 3 处 + 删接口成员。**不碰 FlowEngineService。工作量：小。风险：零（删前 grep 复核）。**

### 🟢 3b · ReadFieldKeys Array/Object 真 bug 修复
- `StageViewProfileResolver.ReadFieldKeys`（:245）只认 `JsonValueKind.Array`，遇 Object 形态（`{"version":2,"fields":[...]}`）返回空 → 该节点 `fieldAccess` 全空 → **审批节点字段权限全部丢失**；而 `CardPresentationResolver.ReadCardSchema`（:409）Array+Object 都处理。口径漂移。
- **修复**：给 `ReadFieldKeys` 加 Object 分支（与 `ReadCardSchema` 对齐），补单测覆盖两种形态。
- **文件**：`Services/StageViewProfileResolver.cs` + 测试。**不碰 FlowEngineService。工作量：小（约一行逻辑 + 测试）。风险：低。是真 bug，优先级高。**

### 🟡 3c · 撤销双份收敛
- 两份口径**实质不同**（非简单包装）：
  - `BatchLifecycleService.RevokeBatchAsync(batchId, operatorId)`（:177）：级联取消卡片 + 凭证红冲 + 行置5 + 标 FIsRevoked；**无** WorkItem/期间预检/事件。调用方 `CardFlowBatchController:347`。
  - `BatchRevokeHandler.RevokeBatchAsync(batchId, operatorId, force, cascadeDeleteFunc)`（:176）：建 WorkItem + WfRevokeLog + 发 `ImportBatchRevokedEvent`/`ImportBatchPurgedEvent`；force 物理删。调用方 `CfImportController:534`。
- **收敛点**：定一份权威撤销，另一处改为调用它；注意 `BatchLifecycleService` 多了"卡片级联红冲 `TryReverseCardVouchersAsync`"，`BatchRevokeHandler` 没有——合并时不能丢这段。
- **文件**：两个服务 + 两个控制器调用点。**不碰 FlowEngineService。工作量：中。风险：中（涉及凭证红冲/物理删，需测试）。**

### 🟡 3d · 处理器 / 工厂收敛
- 4 个 Plugin 薄壳（AlertNotify/InfoRecord/WorkTask 走 `ProcessingPluginBase`、AutoVoucher 走 `BatchPluginBase`）各注入并委托同名 Handler（`IClassificationHandler`）。Plugin 做进度/异常/配置，Handler 做业务。
- 两个工厂并存：`AutoPluginFactory`（生产主路径，`FlowEngineService`/`CfPipelineController`/Express 用）+ `ClassificationHandlerFactory`（仅 `CfDispatchRuleController` 用于派发规则管理接口）。
- 旧 `IAgent`/`IAutoNodeHandler` 接口**已无实现类**（仅 `IAutoPlugin` 注释提及历史）——无需处理。
- **收敛点**：决定派发规则流程是否统一到 AutoPlugin 路径，从而消除 `ClassificationHandlerFactory`；或保留双层委托架构只去掉第二工厂。
- **文件**：4 个 Plugin + 4 个 Handler + 2 工厂 + `CfDispatchRuleController`。**FlowEngineService 只调 AutoPluginFactory、不直接用 Handler，故主路径不动。工作量：中。风险：中。**

### 🟡 3e · 前端步骤3卡片视图 + 动态加签编辑器冻结
- `web/src/views/cardflow/FlowDefinitionEditPage.vue`（~4412 行）：
  - 步骤3「卡片视图」（`activeStep===2`，`CardComponentCatalog` 低代码组件目录常驻 aside，:1857）。
  - 动态加签编辑器 `DynamicApprovalPolicyEditor` 在步骤4 右侧 Drawer 内（:2692，选中 manual 节点时出现）。
- **冻结**：steps 数组移除/隐藏步骤3（:266-283 + :1754 `a-steps`）+ `v-show` 容器（:1857）；Drawer 内 `DynamicApprovalPolicyEditor` 加 `v-if="false"`。组件 `.vue` 文件保留（冻结=隐藏入口）。
- **文件**：仅 `FlowDefinitionEditPage.vue`（约十几行）。**不碰后端。工作量：小-中。风险：低（前端无测试运行器，vite build 兜底）。**

### 🔴 3f · 字段模型双套合一（碰 FlowEngineService）
- `SchemaFieldDefinition`（`Dtos/Responses.cs:399`，5 属性）被 `ConditionEvaluator`/`FlowEngineService:2801`/`FlowGroupService`/`IConditionEvaluator` 用；`CardFieldDefinitionV2`（`Models/Schema/CardSchemaV2Models.cs:23`，属性更全）被展示链用。`CardComponentDefinition` 是 UI 组件维度（另一回事，不并入）。
- **收敛点**：合并前两者为一套扁平字段模型，改 `ConditionEvaluator`/`FlowEngineService` 反序列化签名。
- **文件**：两个模型 + `FlowEngineService`/`ConditionEvaluator`/`FlowGroupService`。**触碰活跃 FlowEngineService。工作量：中。风险：中-高（需 E2E + 既有 98 测试护栏）。**

### 🔴 3g · 条件 evaluator 5 套收敛（碰 FlowEngineService，最大）
- 现存 5 套，operator 命名与 context 协议各不同：
  - `ConditionRuleEvaluator`（主力，JSON 规则树，被 StageRouteResolver/DynamicStagePolicyResolver/PathPreview/ConditionEvaluator 用）
  - `ConditionEvaluator`（字符串 DSL，内部已委托主力；被 `FlowEngineService:1672`/`FlowGroupService:222` 用）
  - `OrchestrationEngineService.EvaluateCondition`（私有，:747，字段用 `op`）
  - `AutoVoucherMatchingEngineV2.EvaluateCondition`（私有静态，:481，`eq/neq/notEmpty`）
  - `ClassificationEngine.BuildWhereClause`（:626，JSON→SQL 参数化 WHERE，`contains/startsWith/isNull/between/in`，被 ClassificationPlugin/CfDispatchRuleController 用，**活跃**）
- **收敛点**：统一到 `ConditionRuleEvaluator` 的 operator 枚举与 context 协议；编排私有、AutoVoucher 私有改调主力；`ClassificationEngine` 的 JSON→SQL 是"内存判定 vs SQL 编译"两种范式，能否统一需专门评估（可能保留但统一规则模型）。
- **文件**：5 处 + 各调用方。**触碰活跃 FlowEngineService。工作量：大。风险：高——最该等工作树稳定 + 单独专项 + 强 E2E 护栏。**
