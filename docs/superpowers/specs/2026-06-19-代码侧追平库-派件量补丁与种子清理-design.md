- 不在本范围：① Finance 品牌版科目 V9 修复、② 凭证规则21 重配（另会话/另 spec）；STG 结构逐表裁决。

- 现行库：47.105.65.51/stotop。

## 背景
1. **派件量功能在库中缺失**：CardFlow 代码 `MigrateV23` = 派件量（建 `STG申通派件日明细` + 规则3006 + 流程2264/版本2265/节点5033），但库 `SYS迁移历史` 的 V23 记录是旧的"网点质控物流完整性明细"（该内容后改名为代码 V50）。因 MigrationRunner 见 V23 已记录即跳过，**代码 V23 的派件量在存量库永不执行**。全新库不受影响（V23 会建派件量、V50 建物流完整性，各一次）。
2. **种子/库口径分裂与废弃残留**（均"以库为准"，库是正确终态）：
   - `Baseline/baseline-reference-data.json` 残留 16 条 `oa:*` SYS功能权限死行（OA 已被 CardFlow 取代，库 oa:* = 0）；BaselineReferenceDataSeeder 按 FID upsert（只增改不删）→ **任何人改 baseline 文件（哈希变化）即把 16 条 oa:* 重新插回库**。
   - baseline.json 缺库已有的 `cardflow:import:validation`(2122)/`cardflow:auto-plugin`(624)/`auto-plugin-rules`(629) → 全新库/baseline 驱动会漏建。
   - `SystemSeeder` 的 gzip+base64 种子 blob 含 28 条 `oa:*` + blob 独有废弃权限（502/503/504/510-517/522/1874 等），库均无 → 全新库/force-init 会复活废弃权限。
   - `WorkflowSeeder` 的 `cardflow.start` 图标写 `AuditOutlined`（死代码段），库/baseline 实为 `FileTextOutlined`。
   - `BasicDataSeeder.EnsureOrgTypes` 仅硬编码 6 个组织类型且带 `Any()` 守卫，缺库已有的 7-9（NETWORK_POINT/FRANCHISE_AREA/LAST_MILE_STATION）。

### 已核实事实
- `MigrateV23` **完全幂等**：建表/4 索引/规则3006/卡片流程2264/流程版本2265/节点5033 全部 `IF NOT EXISTS` 守卫。二次调用安全。
- gzip blob 解码（`Temp/blob_decoded.sql`，638KB/1271 GO 批）：SYS组织架构 320、SYS角色 5、SYS用户角色 4、SYS用户 **500**、SYS功能权限 **441（含 28 oa:*）**、SYS角色权限 **0**（blob 不种角色权限）。
- blob 的组织/角色/用户/用户角色**与库一致**；**唯一与库发散的是 SYS功能权限的废弃行**。
- `SeederHelper.DecodeGzipBase64` = concat chunks → base64 解码 → gzip 解压 → UTF-8；项目**无编码器**，需自写。
- `SYS功能权限` 由 4 处共种（blob + MenuSeeder + RegisterSystemPermissions + baseline）→ 不做"单一真源"大重构。

## 方案

### ③ 派件量 V51 补丁
在 `CardFlowSeeder` 步骤表追加：
```csharp
new(51, "派件量补丁：存量库补建 STG申通派件日明细+规则3006+流程2330（V23在存量库被网点质控占用未执行）", MigrateV23),
```
- 复用幂等 `MigrateV23`，**不复制内容**。存量库：V51 新建派件量三件套；全新库：V23 建、V51 全 `IF NOT EXISTS` 空跑。
- 保持 `ValidateSteps` 连续性（当前 max=50 → 加 51 连续）。`MigrateV23` 内容不动。
- **风险段（不在本 spec 解决）**：规则3006 的 `excelColumn`/`F组织ID=192`/`fileNamePattern` 为占位，真正导入前须按真实派件文件 row1 表头核对；3006 沿用旧 3xxx 编号方案（与库 live 18-24 编号体系统一另议）。

### ④a baseline.json 清理
`src/STOTOP.WebAPI/Data/Seeders/Baseline/baseline-reference-data.json`：
- **删** SYS功能权限 表里 16 条 `oa:*`（FID 500/501/505-509/518-521/525-529）。
- **补** SYS功能权限 三行：`cardflow:import:validation`(FID2122,按钮,父157)、`cardflow:auto-plugin`(FID624,菜单,父150)、`auto-plugin-rules`(FID629,菜单,父150)，字段对齐现库实际值（实现时从库 `SELECT` 取准确字段）。
- baseline 文件改动会使其 SHA256 变化 → 下次启动 BaselineReferenceDataSeeder 重跑 upsert（幂等，库已有这些行，无副作用；oa:* 已从文件移除故不再复活）。

### ④b gzip blob 彻底清理
写 Python 工具 `Temp/blob_rebuild.py`（或落 scripts/）：
1. 从 `SystemSeeder.cs` 提取 `SqlChunk0..12` → 解码（concat→base64→gzip→UTF-8）。
2. 计算待删集合 = **blob 的 SYS功能权限编码 − 现库 SYS功能权限编码**（= 28 oa:* + blob 独有废弃 FID）；从解码 SQL 中删除这些 `INSERT INTO [SYS功能权限] ...` 行（按 F编码/FID 精确匹配，保留 GO/IDENTITY_INSERT 结构完整）。
3. 重新 gzip+base64，按原 chunk 切分长度生成新的 `SqlChunk0..N` 常量，替换 `SystemSeeder.cs` 中对应常量。
4. **组织架构/角色/用户/用户角色段字节级保持不变**（仅删功能权限废弃行）。
- 验证：重生 blob 解码后 `SYS功能权限` 编码集合 ⊆ 现库（无 oa:*、无 502-522/1874）；其余表段与原解码逐行一致（diff 仅含被删行）；解码→重编码→再解码可往返一致。

### ④c WorkflowSeeder
`cardflow.start` 的 `F图标` `AuditOutlined` → `FileTextOutlined`（与库/baseline 一致）。

### ④d EnsureOrgTypes
`BasicDataSeeder.EnsureOrgTypes` 补 3 个组织类型 FID 7-9：NETWORK_POINT(快递网点,层级3)、FRANCHISE_AREA(承包区,层级4)、LAST_MILE_STATION(末端驿站,层级4)，字段对齐库/baseline。

## 特别标注（不在本 spec 改）
- blob 含 **500 行真实用户（含密码哈希）**，与库一致，故"以库为准"保留；但"整库用户当种子提交进源码"是独立的安全/隐私隐患，建议另案处理（不属三方同步范畴）。
