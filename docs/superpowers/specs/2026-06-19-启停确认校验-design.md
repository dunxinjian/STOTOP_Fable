# 启停确认 + 校验（P2）— 设计

- 日期：2026-06-19
- 范围：科目启停 switch 加停用确认 + 后端停用校验（防静默漏记、余额消失）。
- 优先级：P2（危害控制）
- 工作区：worktree `finance-p2d`

## 1. 问题

启停 a-switch 即时生效、无确认（`handleStatusChange` 直接 `toggleAccountStatus`）；后端 `ToggleStatusAsync`（`AccountService.cs:294-302`）只翻转 `FEnableStatus`、**零校验**。危害：停用被自动凭证规则引用的科目 → `VoucherAutoService` 静默漏记；停用带余额科目 → 期初合计余额凭空消失（`GetInitialBalances` 按 `FEnableStatus==1` 过滤）。对比 `DeleteAsync` 有凭证/子级双校验，危害相近的停用反而零确认零校验。

## 2. 修复设计（前后端双层）

### 2.1 后端 ToggleStatusAsync 停用校验
仅在「启用→停用」（`account.FEnableStatus == 1`）时校验，启用（停用→启用）无害不校验。任一命中则抛 `InvalidOperationException` 拒绝（复用 `DeleteAsync` 的检查模板）：

```csharp
    public async Task<bool> ToggleStatusAsync(long id)
    {
        var account = await LoadAccountAsync(id);
        if (account == null) return false;

        // 停用时校验（启用无害，不校验）：被凭证引用/有启用下级/有余额则拒绝
        if (account.FEnableStatus == 1)
        {
            var hasVoucher = await _voucherEntryRepository.Query()
                .IgnoreQueryFilters()
                .AnyAsync(e => e.FAccountId == id);
            if (hasVoucher)
                throw new InvalidOperationException("该科目已被凭证引用，无法停用");

            var hasEnabledChildren = await _accountRepository.Query()
                .AnyAsync(a => a.FParentId == id && a.FEnableStatus == 1);
            if (hasEnabledChildren)
                throw new InvalidOperationException("该科目有启用的下级科目，无法停用");

            var hasBalance = await _balanceRepository.Query()
                .AnyAsync(b => b.FAccountId == id
                    && (b.FBeginDebit != 0 || b.FBeginCredit != 0 || b.FEndDebit != 0 || b.FEndCredit != 0));
            if (hasBalance)
                throw new InvalidOperationException("该科目存在余额，无法停用");
        }

        account.FEnableStatus = account.FEnableStatus == 1 ? 0 : 1;
        account.FUpdatedTime = DateTime.Now;
        await _accountRepository.UpdateAsync(account);
        return true;
    }
```

> `AccountController.ToggleStatus` 已 `try/catch InvalidOperationException → ApiResult.Fail`？需确认；若无则补（参照 Create/Delete），使拒绝消息能到前端。

### 2.2 前端 handleStatusChange 停用确认
停用前 `Modal.confirm`（启用直接）；失败由请求拦截器弹后端消息（与本文件既有约定一致，catch 仅 console.error）。

```ts
async function handleStatusChange(row: any, val: boolean) {
  if (!val) {
    Modal.confirm({
      title: '确认停用',
      content: `确定要停用科目「${row.name}」吗？被凭证引用、有启用下级或有余额的科目将无法停用。`,
      okType: 'danger',
      onOk: () => doToggleStatus(row, val),
    })
    return
  }
  doToggleStatus(row, val)
}

async function doToggleStatus(row: any, val: boolean) {
  try {
    await toggleAccountStatus(row.id)
    message.success(val ? '已启用' : '已停用')
    loadData()
  } catch (error) {
    console.error('状态切换失败:', error)
  }
}
```

a-switch 受控（`:checked="record.isEnabled"`），取消/后端拒绝时不调 `loadData`，switch 自动回弹到原状态。

## 3. 测试

- **后端集成**（InMemory，CreateService 需传真 `Repository<FinVoucherEntry>`，ToggleStatus 用到）：
  - 停用被凭证引用科目 → 抛"凭证引用"、状态仍启用；
  - 停用有启用子级 → 抛"下级"；
  - 停用有非 0 余额 → 抛"余额"；
  - 正常停用（无引用/子级/余额）→ 通过（FEnableStatus=0）；
  - 启用（0→1）即使有凭证也不校验 → 通过。
- 前端：`npx vite build` + 手动（停用弹确认、被引用停用拒绝并回弹）。

## 4. 非目标（YAGNI）

- `toggle → setStatus(enabled)` 幂等化（另立）。
- 封存第三态（另立）。
- 余额科目的"先结转再停用"流程（仅拒绝，不提供结转）。
