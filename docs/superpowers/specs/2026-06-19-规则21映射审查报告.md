- 审查对象：`Temp/rule21_mapping_v2.json`（草案，groups85 + gapBC[24] + gapA[3]）
- 权威基准：`Temp/brandcoa_parsed.json`（品牌版科目，acctSet2=太仓美申，fid 700001–700427）
- 原始组：`Temp/rule21_groups_full.json`（85组 exactCodes/oldAccountCode）
- 核查项数：109（groups85 共85 + gapBC 24）；逐条做了 fid 实存/code 一致/name 一致/isLeaf/会计方向/收入支出与科目类别自洽 六项机检 + 同码异义/折叠/方向人工复核

## 总体结论

技术层面草案“干净”：**109 条贷方目标 fid 全部实存于 acctSet2，code/name 与库一一对应，全部 isLeaf=1，无编造或失效 fid**；借方固定 220201 (fid 700125, isLeaf=1) 正确。

但会计口径层面存在**系统性错账风险，未经财务确认不可落库**，分四类：

1. **收入↔成本错位（最严重，11 组）**：amountField=收入额 却映射到“营业成本”科目，会把收入记进成本，同时虚增成本、漏记收入。
2. **同码异义漏网（1 组致命 + 多组已被草案标注）**：#11 code 未变（50010702→50010702）但品牌版该 code=“三件私了”与业务“按需派送考核激励”不符。
3. **匹配键冲突（结构性，11 个 feeCode 被多组占用）**：规则21 按费用编码匹配，同一 exactCode 解析到 2–3 个不同科目，引擎无法确定性命中 → 抢占/漏记/重复。
4. **business_direction 自洽性（3 组收入/成本孪生方向打架 + 1 组折叠错位）**。

## 一、收入↔成本错位（须财务拍板）

机检：`incomeOrExpense=收入额` 但目标科目 `category=营业成本`。这些组取的是 `F发生额收入`（网点向总部付费=对总部为收入），却被映射到成本叶。**借 220201 / 贷 营业成本** 的分录在会计上等价于“冲减成本”，而非确认收入——若财务本意是收入，则收入被全额漏记、成本被全额低估。

根因：品牌版**收入侧无对应叶**（增值服务收入 500104 仅剩 50010401 网点线路跑车一叶；集包/中转/全网出港/环保袋/经营支持/网点赋能/客服服务在收入侧均无叶，只在成本侧 5401 有叶）。草案被迫“借用成本叶”。两条候选必须财务二选一：
- 方案甲（收入冲减成本）：维持记成本叶，承认这些“受款”本质是成本分摊回收（冲减成本）。则借贷应为 **借 220201 / 贷 5401x（红字/负数冲减）**，且 business_direction 取 IN/CMB 不变；现草案是正数贷成本，方向与“冲减”相反，需复核符号。
- 方案乙（确认收入，新增叶）：在品牌版收入侧 500103/500104/500105 下**新增对应收入叶**（如“集包受款/中转受款/增值受款”），再改记。落库前须先补种子科目。

> 注：#28 客服赔受款（收入）已映到收入叶 50010701 客服受款（属营业收入），机检未报错；但其与 #69 客服赔款（成本 54010701）是“受款 vs 赔付”一进一出，需财务确认“受款”确为收入而非冲减赔付成本。

## 二、同码异义

### 2.1 致命漏网：#11 按需派送考核激励

- 组名/性质 = **按需派送考核激励**（考核激励，应属 500106 考核激励族）。
- code 平移看似“安全”，实为同码异义最隐蔽形态：旧账套 50010702 与品牌版 50010702 含义完全不同。

## 六、business_direction 自洽性

同业务收入侧/成本侧孪生组方向应一致，下列打架须统一：
- **小件员权益**：收入#37 IN vs 成本#80 OUT。派费派送性质应统一（建议 IN，派费=进港派送）。
