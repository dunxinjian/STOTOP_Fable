# 损益取数修复（P1-2）— 设计

- 日期：2026-06-19
- 范围：修复「损益类科目取数硬匹配 `FCategory == "损益"`」导致的利润表损益恒 0、期末结账不结转损益。
- 优先级：P1（活路径、结构性必错、纯代码、无数据迁移）

## 1. 问题与事实依据

### 1.1 根因
科目种子中**损益类科目的 `F类别`（FCategory）100% 是子类名**——营业收入 / 营业成本 / 营业税金及附加 / 期间费用 / 其他收益 / 其他损失 / 所得税费用 / 以前年度损益调整。大类名「损益」在科目种子里出现 **0 次**（统计 `src/STOTOP.WebAPI/Data/Seeders/FinanceSeeder.cs` 全部 `FIN科目` INSERT 得证）。

而下游两处取数硬匹配大类名 `FCategory == "损益"`：
- `ReportService.cs:1021`（`CalculateAccountAmounts`）：损益科目走 `else` 置 `amount = 0`。
- `AccountPeriodService.cs:192`（期末结账）：`profitAndLossAccounts` 集合恒为空。

### 1.2 受影响面（全集，已穷尽）
全局检索 `== "损益"` 仅命中上述两处，无第三处。

- **两张利润表全部经过 `CalculateAccountAmounts`**：
  - `GetProfitStatementAsync` → `ReportService.cs:398/399`
  - `GetSmallEnterpriseProfitStatementAsync` → `ReportService.cs:923/924`
  - 二者再用 `GetAmountByCodePrefix("5001")` 等从该字典取数，而字典里损益科目金额全为 0 → **两张利润表的营业收入/成本/费用/利润各行全为 0**。
- **期末结账**（`AccountPeriodController.cs:150` 的 `Close` 端点）：损益集合为空 → 结转循环不执行 → `closingEntries` 为空 → `if (closingEntries.Count > 0)` 为 false（`AccountPeriodService.cs:264`）→ **不生成任何结转凭证**；期间却照常被标记已结账。后果：损益科目余额永不清零、本年利润 3103 恒为 0、12 月「本年利润→未分配利润 310405」连带失效。

### 1.3 不受影响、保持不动
收入 vs 费用的细分仍按 `FBalanceDirection`（贷=收入、借=费用）。损益科目种子方向已逐条核对正确（5001 收入=贷、5401 成本=借、5601~5603 费用=借 等），本次不动。

## 2. 修复设计

### 2.1 判定口径
**按类别集合判定**：科目属于「损益子类集合」即视为损益科目。集合中额外纳入大类名 `"损益"`，兼容历史/其他账套可能存在的以大类名落库的脏数据。无需数据迁移。

> 口径取舍：未采用「FCode 首位 5」是为与现有「按 FCategory」的代码风格一致、语义直观；集合含 `"损益"` 已覆盖历史脏数据，健壮性足够。

### 2.2 新增单一真源
新建 `src/STOTOP.Module.Finance/Constants/FinAccountCategory.cs`（与现有 `Constants/AuxTypes.cs`、`BusinessDirection.cs` 同风格、同命名空间约定）：

- `ProfitLossCategories`：损益子类集合，类型用数组/`IReadOnlyList<string>` 以便 EF Core 翻译为 SQL `IN`。内容 =
  `{ "损益", "营业收入", "营业成本", "营业税金及附加", "期间费用", "其他收益", "其他损失", "所得税费用", "以前年度损益调整" }`
  （与 `AccountService.GetSubCategories("损益")` 现有数组保持一致）。
- `IsProfitLoss(string? category)`：成员判定，`null`/空返回 false。

### 2.3 三处替换
1. `ReportService.cs:1021`：`if (account.FCategory == "损益")` → `if (FinAccountCategory.IsProfitLoss(account.FCategory))`。一次修好两张利润表。
2. `AccountPeriodService.cs:192`：`.Where(a => a.FCategory == "损益" && a.FIsLeaf == 1 && a.FAccountSetId == accountSetId)` → 用 `FinAccountCategory.ProfitLossCategories.Contains(a.FCategory)` 替换 `a.FCategory == "损益"`（EF 翻译为 `IN`）。修好结账损益结转。
3. `AccountService.cs:87`：`GetSubCategories` 的「损益」分支返回值改为引用 `FinAccountCategory.ProfitLossCategories`，避免新造双真源（该集合本就含 `"损益"` 自身，与现有兜底语义一致）。

## 3. 测试（TDD，先红后绿）

实现首步先勘察 `tests/STOTOP.Module.Finance.Tests` 现有基建（DbContext 构造方式、既有 Service 测试模式），按其约定落测试。覆盖：

- **结账**：构造损益末级科目（收入=贷、费用=借）+ 本期已审核凭证发生额 → 调结账 → 断言：① 生成「转」字结转凭证；② 损益科目本期被冲平；③ 本年利润 3103 分录借贷=收入净额/费用净额。
- **利润表**：同一组数据 → `GetSmallEnterpriseProfitStatementAsync`（及/或 `GetProfitStatementAsync`）→ 断言「营业收入」行金额 = 实际收入且非 0。
- **回归保护**：断言「损益」大类名脏数据也被识别（集合含 `"损益"`）。

## 4. 数据与运维注意

- **无需数据迁移**：种子数据本身正确，错在消费端取数；集合含 `"损益"` 兼容历史。
- **已被错误「结账」过的历史期间不会被本次自动修正**：这些期间损益从未结转、3103 为 0。补救需「反结账 → 重新结账」，属运维动作，**不在本次范围**。修复仅保证此后的结账与利润表正确。
