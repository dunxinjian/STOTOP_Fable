## 3. 取数口径

### 跨期合成规则（核心）

| 指标类型 | 跨期合成 | 多网点合并 | 例 |
|---|---|---|---|
| **率值类** | 区间**末日快照** | 末日当天**件量加权** | `F当天及时签收率`、`F48h签收率`、`F一频次出仓及时率`、`F滞留率`、`F虚签投诉率`、`F遗失率ppm` |
| **累计量/金额类** | 区间**累计**（SUM over 区间×网点） | 直接相加 | `F考核金额合计`、事件计数、`F遗失量`、各域 `F*预估考核金额` |
| **趋势图** | 日粒度原值逐日打点（不合成），独立于 KPI 卡 | 按日合并（率件量加权 / 量相加） | 近30天多指标折线 |

> KPI 卡 = "区间末日的状态 + 期内累计的量"；趋势图 = "期内每天走势"。单网点筛选时无合并问题；多网点/全组织时率值才需件量加权（件量取当日 `F日均出港量` 或对应量列作权重，缺权重列时退算术平均并在实现注释标注）。

### 字段映射（已核实实体字段名）

**视图1 网点总览**（源 `QlShentongNetworkDailyMetric`，域分布源事件表）
- 签收率：主 `F当天及时签收率` / 副 `F48h签收率`
- 出仓及时率 `F一频次出仓及时率`；交货滞留率 `F滞留率`；积压量·倍数 `F积压倍数`（+ `F日均出港量`/`F超3天积压量`等辅助）；遗失率ppm `F遗失率ppm`；虚签率 `F虚签投诉率`；考核金额合计 `F考核金额合计`
- 考核金额按域构成：`F出仓预估考核金额` + `F滞留预估考核金额` + `F派送预估考核金额` + `F签收率考核金额`（各域期间累计）
- 质量域问题分布：事件表 `GROUP BY F质量域` 期间累计计数

**视图2 员工质量**（源 `QlShentongEmployeeDailyMetric` + 事件表）
- 红黑榜 5 维：问题件（事件计数，按 `F员工工号` 聚合）、客诉、虚签、超时、考核金额（后四维取员工日指标对应汇总列）
- 21 指标明细：员工日指标逐列（派件/派签/上门/客诉/虚签/质检/超时/违规/`F考核金额合计`）
- 单员工钻取：事件表 `WHERE F员工工号=? ORDER BY F业务日期`，JOIN 字典取归一问题名

**视图3 问题件追踪**（源 `QlShentongQualityEvent` JOIN `QlShentongProblemDict`）
- 明细列：`F运单号`、`F网点编码/F网点名称`、`F员工工号/F员工姓名原文`、`F质量域`、`F问题类型名称`、`F严重度`、`F是否考核件`、`F考核金额`、`F电商平台`、`F业务日期`
- 筛选维度：承运商、`F质量域`、`F电商平台`（平台）、网点、`F员工工号`、日期、`F严重度`
- 多域命中：同一 `F运单号` `COUNT(DISTINCT F质量域) >= 2` 即高亮
- 待认领灰显：`F员工匹配状态 ∈ {0,3}`（未匹配/启发式未绑）

> 字段订正记录：事件表运单字段为 **`F运单号`**（非 `F运单`）；视图3"平台"维度对应 **`F电商平台`**。

## 4. 后端 API 设计

路由前缀 `/api/quality/carrier-dashboard`，权限 `CarrierQualityView`（认领相关沿用 `/api/quality/unify`，权限 `CarrierMasterDataClaim`）。

公共查询参数：`carrier`（默认"申通"）、`from`/`to`（日期区间，默认近30天）、`networkCode?`。

**视图1**
- `GET network/kpi` → 7 KPI（率值末日快照 / 量金额期间累计），返回各指标值 + 主副签收率
- `GET network/trend?metrics=` → 近30天多指标日粒度序列
- `GET network/domain-distribution` → 域问题分布（事件表 group by 质量域）
- `GET network/fee-by-domain` → 考核金额按域构成（四域预估金额期间累计）

**视图2**
- `GET employee/rank?dimension={problem|complaint|fakesign|timeout|fee}&topN=` → 正反榜
- `GET employee/metrics?page=&size=` → 21 指标明细分页
- `GET employee/{empNo}/timeline` → 单员工事件时间线

**视图3**
- `GET events?{filters}&multiDomainOnly=&pendingOnly=&page=&size=` → 事件明细分页（带 isMultiDomain / isPending 行标志）
- `GET events/export?{filters}` → 后端全量 CSV（BOM + csvEscape + 时间戳文件名）

**认领角标**
- 复用 `GET unify/pending-employees` 的 total 作角标计数

## 5. 视图详细设计

### 视图1 · 网点总览（网点 × 期）
顶部统一筛选（承运商占位 + 日期范围近30天 + 可选网点）→ 7 KPI 条带（方向性着色，签收率主副双值）→ 近30天多指标趋势（ECharts 折线，主色从 `themeStore` 取**真 hex**）→ 质量域问题分布（饼/柱）→ 考核金额按域构成（堆叠柱）。可选网点明细表（每网点一行期内指标）。

### 视图2 · 员工质量（工号 × 期）
红黑榜 5 维各出正反榜 TopN → 21 指标明细表（分页50、合计行、`formatNum`/`formatRate(null→'—')`/金额¥+2位）→ **单员工钻取**：点工号开抽屉/展开，显示该工号名下全部质量事件时间线（细到工号的人物关联实现路径）。

### 视图3 · 问题件追踪（运单 × 域）
事件明细表 + 多维筛选 → 多域命中重点件高亮（≥2 域）+ "仅看重点件"筛选 → 待认领灰显 + "仅看待认领"筛选 → 后端全量 CSV 导出。

### 配套 · MasterDataClaim（认领页）
后端全现成：待认领员工清单（姓名原文×网点 + 计数 + SuggestStatus 启发式建议）→ 维护员工别名 `ExpSalesmanAlias` / 网点主数据 → `POST unify/rematch` 触发重跑回填。三视角导航处显示待认领总数角标引导清账。

## 6. 一期边界（明确不做）

不碰 `DetectionService`/`DispatchService` 空壳、不写 `QlPerformance`、不自算 KPI 公式、不派钉钉/待办、不做奖惩、不做阈值告警规则、不接其他承运商（维度占位但物理只申通）。这些属二期（父 spec §11.7/§12）。

## 7. 错误 / 空数据 / 性能

- 空数据：率值 KPI 显 "—"（`formatRate(null→'—')`），图表空态，不报错
- 分页：明细表 50/页，`showSizeChanger/showQuickJumper/showTotal`；排序 null 沉底
- 性能：四表复合唯一键 `(FOrgId,F承运商,F业务日期,…)`，查询走索引前缀；趋势限近30天数据点；导出后端流式避免前端内存
- 组织切换：`watch(orgContextStore.orgSwitchVersion)` 重取；请求头 `X-Org-Context`/`X-AccountSet-Id` 由 `api/request.ts` 自动注入
- 主题色：ECharts 不解析 `var()`，主色从 `themeStore` 读真 hex 传入；样式走 `var(--color-*)` 令牌禁裸 hex
