# 品牌版科目 V9 落库修复 — 设计 spec

- 日期：2026-06-19
- 范围：**仅 ① Finance V9 修复**（让 `FinanceSeeder.MigrateV9` 在 dev 测试库可安全执行）。
- 不在本范围：② 凭证规则21 按品牌版重配（另起 spec，需会计口径的"费用名称→品牌版科目"业务映射）；③ 派件量 V23 补丁；④ 种子/baseline 清理；STG 结构逐表裁决。
- 关联：`Temp/三方差异比对报告.md`、记忆 `stotop-three-way-sync-diff`、`stotop-accounting-amoeba-analysis`、`shentong-hq-tx-import-test`。

## 1. 背景与问题

`FinanceSeeder.MigrateV9`（品牌版科目覆盖）当前实现：
```csharp
ExecSql(ctx, @"DELETE FROM [FIN科目] WHERE [F账套ID] IN (1,2);");
InsertBrandAccounts(ctx);   // 重灌品牌版 909 行（账套1=482/账套2=427，含各账套真实 1002/1012）
```
在测试库（47.105.65.51/stotop）执行时**被外键阻塞并整事务回滚**：
```
DELETE FIN科目 与 REFERENCE 约束 FK_FIN资产类别_折旧科目ID 冲突
```

### 已核实事实
- 引用 `FIN科目` 的外键共 5 个：`FK_FIN凭证分录_科目ID`、`FK_FIN科目余额_科目ID`、`FK_FIN辅助核算余额_科目ID`、`FK_FIN凭证模板分录_科目ID`、`FK_FIN资产类别_折旧科目ID`。
- 账套1/2 在前 4 个（事务性）外键的引用数**全为 0**（凭证/分录/科目余额/辅助核算余额/凭证模板分录均 0）。
- **唯一阻塞者 = `FIN资产类别`**：6 行固定资产类别（房屋建筑/生产设备/器具工具家具/运输工具/电子设备/其它固定资产）的 `F对应折旧科目ID` **全部指向账套1 科目 `1602 累计折旧`(旧 FID 200550)**。
- `FIN资产类别.F对应折旧科目ID` **可空（IS_NULLABLE=YES）**。
- `FIN科目` 有唯一约束 `IX_FIN科目_编码_账套ID (F编码,F账套ID)` → 旧 1602 与新 1602 不能并存，**必须先删后插**。
- 品牌版 `1602 累计折旧`：账套1=FID 600031、账套2=FID 700031（F辅助核算='department'）。
- V9 上次执行**未写入 `SYS迁移历史`**（仅 `SYS迁移执行日志` 留 1 条 Failed）→ 原地改 `MigrateV9`，下次启动自动重跑修正版，**无需新版本号**。

## 2. 方案（捕获→置空→重指）

删账套1/2 科目前，先解开 `FIN资产类别` 的折旧科目外键（该列可空，无需动 DDL），重灌品牌版后按业务键 `(F编码, F账套ID)` 重指到新科目 FID。通用实现（不写死 6 行）。

### 修复后 MigrateV9 流程（MigrationRunner 单事务内顺序执行）
1. **空账套守卫（fail-safe）**：统计账套1/2 在 4 个事务性外键表（`FIN凭证分录`/`FIN科目余额`/`FIN辅助核算余额`/`FIN凭证模板分录`）的引用行数；**任一 >0 即 `throw InvalidOperationException` 中止**，绝不删除挂有账务数据的科目。资产类别不计入阻断项（参考数据，走步骤2重指）。
2. **捕获并解开资产类别 FK**：将 `FIN资产类别` 中折旧科目落在账套1/2 的行记录 `(资产类别FID, 折旧科目F编码, 折旧科目F账套ID)` 到临时表 `#dep`；`UPDATE [FIN资产类别] SET [F对应折旧科目ID]=NULL WHERE FID IN (SELECT 资产类别FID FROM #dep)`。
3. **删旧科目**：`DELETE FROM [FIN科目] WHERE [F账套ID] IN (1,2)`（保留现有逻辑）。
4. **重灌品牌版**：`InsertBrandAccounts(ctx)`（保留现有，含各账套真实 1002/1012）。
5. **按业务键重指**：
   ```sql
   UPDATE ac SET ac.[F对应折旧科目ID] = k.FID
   FROM [FIN资产类别] ac
   JOIN #dep d ON d.资产类别FID = ac.FID
   JOIN [FIN科目] k ON k.[F编码] = d.折旧科目F编码 AND k.[F账套ID] = d.折旧科目F账套ID;
   ```
   当前 6 行（编码 1602/账套1）在品牌版存在(FID 600031)→全部命中。若某编码在品牌版已消失→该行保留 `NULL` 并日志告警（不中止；该列可空，避免阻断整个迁移）。

### 实现要点
- 全部步骤在 `MigrateV9` 内、由 `MigrationRunner.RunMigrations` 的步骤事务包裹，任一步失败整体回滚、不写版本号。
- 临时表 `#dep`（或表变量 `@dep`）在同一连接/会话内有效即可；用 `SeederHelper.ExecuteRawSql`（纯 ADO，绕 EF String.Format，避免大括号转义问题）。
- 守卫中止信息要可读（列出哪个表有多少引用），便于在生产/非空库上理解为何拒绝执行。

## 3. 执行与验证
- **执行**：以 Development 环境启动后端（`dotnet run --project src/STOTOP.WebAPI`）触发 `MigrateAll`→Finance V9。账套空、已确认安全。完成后停后端（恢复 dev 原状）。
- **只读核对**（用 `Temp/dbq.py`）：
  - `FIN科目` 总数 = 910 = 品牌版 909（账套1=482、账套2=427）+ 账套0=1 孤儿（V9 只删账套1/2，孤儿仍在，不在本 spec 处理）。
  - 账套1/2 科目 FID 段为 6xx/7xxxxx。
  - `FIN资产类别` 6 行 `F对应折旧科目ID` = 品牌版账套1 1602(FID 600031)，无残留 NULL。
  - `SYS迁移历史` Finance 最高版本 = 9。
- **测试取向**：dev 测试库直跑（账套空，零事务数据）。无需隔离库（隔离库全新建库验证属 ② 之前的更大计划，已超本 spec）。

## 4. 风险与回滚
- **连锁（重要，写入风险段，不在本 spec 实现）**：V9 成功后凭证规则21 的 88 个 accountId 失效→凭证生成空窗，须紧跟 **② 规则21 品牌版重配**（另起 spec）。dev 账套空、无实时凭证生成，空窗当前无实际危害。
- **回滚**：V9 任一步失败由事务自动回滚（保持网点版现状）；如需主动还原网点版，可从 git 历史/baseline 的网点版科目定义在 dev 库重灌。
- **账套0 孤儿科目** `54010403 出港补贴派费`(FID 201330)：V9 的 DELETE 仅作用 `F账套ID IN(1,2)`，不动账套0；保留现状，清理裁决归入 C 组（另议）。
