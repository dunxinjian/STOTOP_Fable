# CardFlow 阶段1：清死代码 实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: 用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 按任务逐条实现。步骤用 `- [ ]` 复选框跟踪。
> 所有 commit 末尾追加：`Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`

**Goal:** 删除 CardFlow 模块中确证零调用的死文件并清理其 DI 注册，冻结（隐藏路由入口）3 组零数据/未完工的前端子系统，使主干干净、上手路径唯一——不改任何在生产跑的导入主链行为。

**Architecture:** 纯减法。后端逐项「先 Grep 复核零调用 → 删整份文件 → 删 `CardFlowModuleExtensions.cs` 对应 DI 行 → `dotnet test` 验证编译+单测全绿 → commit」。前端移除编排/下载自动化/质量规则的路由对象（`.vue` 组件文件保留在磁盘上，即「冻结=保留代码」），`vite build` 兜底。每个任务独立 commit，可单独回滚。

**Tech Stack:** .NET（C#，`STOTOP.Module.CardFlow`）、xUnit + EFCore.InMemory（`tests/STOTOP.Module.CardFlow.Tests`）、Vue 3 + Vite（`web/`）、Windows / PowerShell。

## 与 spec 的范围修正（基于第一手 grep 实证，3 处收紧）

本计划比 spec 第4/5节更保守，原因是 grep 复核发现：

1. **`BatchTriggerService` 的 `[Obsolete]` 段 / `BatchJobKind` 退化分支 → 移出阶段1，改到阶段3。** 因为 `BatchJobKind` 与 `ConfirmStagingAndFanOutAsync` 被**活控制器**调用（`CfImportController.cs:534/992/1312/1593`、`CardFlowBatchController.cs:332`、`BatchJobProcessorService.cs:117-127`），不是零调用，删除有风险，须在阶段3 收敛时带护栏处理。
2. **`QualityAnalysisPlugin.ValidateRowBuiltin`（活文件里的死私有方法）+ `ITransformEngine.Preview`（冻结的转换引擎内部）→ 移到阶段3。** 二者都在「保留/冻结」的活文件内部，阶段1 坚持「只删整份死文件」，不切活文件内脏，避免无测试网下的内联编辑风险。
3. **`ClassificationHandlerFactory` 必须保留**（被 `CfDispatchRuleController.cs:22` 注入）；阶段1 只删 `DispatchRouter`（确认零调用）。
4. 前端「动态加签编辑器 / 低代码组件目录 Tab」是嵌在 `FlowDefinitionEditPage.vue`(4412 行) 里的内嵌组件，非路由，阶段1 不动；改到阶段3 的前端专项。阶段1 前端只摘**路由级**入口。

> 这些被移走的项**不丢失**，已登记在 spec 的阶段3 / 桶C，待真实流 + E2E 护栏就位后处理。
